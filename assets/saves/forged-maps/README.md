# Forged map saves — real states the console authored, at maps you can't easily reach

Save fixtures for **working on the map screen**, generated by `scripts/emu/forge_map_save.py`.

## What makes these different from `natural-clean/`

`natural-clean/BaseSAV.sav` is a real save, but it stands in **Pallet Town** — a small, quiet map
with one script trigger and almost nothing else on it. Good for a baseline; a poor bench for map
work, because most of what the map screen has to draw simply isn't there.

These are the **busy** maps. Same authenticity, more to look at.

## They are NOT hand-built, and that is the whole point

⚠️ Poking a map id into a save produces a **chimera** — the new map's id with the old map's Area
block (dimensions, tileset pointers, scripts, sprites, warps, wild tables) — and a real console
**hard-wedges about 100 frames after Continue**. See
[`../../notes/reference/forged-saves.md`](../../notes/reference/forged-saves.md).

So we don't build the state. **The game does.** `forge_map_save.py` boots the real ROM on BaseSAV,
rewrites the front door's destination in live WRAM, walks the player through it, and lets the
engine's own map loader run — every header byte, pointer, sprite slot, connection and wild table
authored by the code they belong to. Once the arrival settles it does exactly what
`SaveSAVtoSRAM` does and reseals the checksum. The result is byte-for-byte what saving there
in-game would produce.

That is why the map "is generated proper": nothing here was reasoned out by us.

## Using one

```
PokeredSaveEditor.exe --sav assets\saves\forged-maps\Route12.sav --screen maps
```

## Adding one

```
tmp\emu-venv\Scripts\python.exe scripts\emu\forge_map_save.py --map <id>
```

It caches to `tmp/emu/map-saves/map<NNN>.sav` (git-ignored); copy it here under a **readable name**
and add a row above saying *why that map*. Needs the ROM at `tmp/emu/rom.gb` — **never commit the
ROM**. The 8 warpless routes are reached by the edge hop automatically.
