### 2026-07-16 -- The PyBoy wedge, root-caused: hang verdicts, tree-kills, and the shared forge

The "tons of processes and everything hangs" mystery is solved, reproduced, and defended against at
every layer. Three stacked causes: **(1)** a map-id-only forged save is a *chimera* — the new map's id
wearing the old map's Area block — and the console **hard-crashes ~100 frames after Continue**, where
the runaway CPU executes `STOP`, the clocks halt, and **PyBoy's `tick()` never returns** (a full core,
forever); **(2)** killing the wedged child leaked its interpreter, because the venv `python.exe` is a
launcher — only a process-**tree** kill takes the spinning PyBoy with it; **(3)** the time-capped
interactive shell turned both into a leak spiral. So: `tst_flag_scenarios` is now **wired into CMake
and live** (one scenario per process; a timed-out scenario is recorded as a **"hang" verdict** and the
batch carries on — only the control gate hard-fails), the forge + checksum-reseal recipe is extracted
into one shared `scripts/emu/forge_save.py` (map/coords/named flags/raw pokes, CLI + importable), and
`scripts/emu/drive_session.py` gives interactive emulator driving the correct shape: one PyBoy, one
owned child process, NDJSON over pipes, frame-bounded everything. The 2026-07-15 "forge-onto-any-map
PROVEN" claim is **overturned** in the notes — same-map forges are safe; the consistent cross-map
forge is future work.

### 2026-07-15 -- Layers: "Warp tiles" restored — the tile-trait and the warp objects are DIFFERENT things (0.40.3-alpha)

Follow-up to 0.40.2. Twilight: the warp tile-trait "has to come back — it was ok as it was." She's
right, and the earlier removal was too hasty. There genuinely are two different warp things and both
belong:

- **Warp tiles** (the tile-trait, `1<<3`): which tile *graphics* are warp-capable on a tileset (doors,
  stairs, cave mouths, warp pads). From `data/tilesets/warp_tile_ids.asm`; `IsWarpTileInFrontOfPlayer`
  checks the tile you walk into against the tileset's list to know a warp *can* fire there. A tileset
  fact, shared by every map using that tileset.
- **Warps** (the object layer, Game View): the save's warp *list* — the coordinates and destinations,
  the draggable objects. Map state.

So this restores the tile-trait to the **Tiles** group, relabelled **"Warp tiles"** (and the door
description clarified: a door is a passable tile you cross to reach a warp, not the warp itself) — the
name change is what kills the original "two Warps" confusion, without losing a real layer. Defaults
unchanged (Tiles all off, Game View warps-object on). `tst_map_layers` back to expecting the nine tile
overlays (incl. Warp tiles) — 12/12; `tst_map` 27/27.

### 2026-07-15 -- Layers: the "Components" group becomes "Tiles", and warps move to where they belong (0.40.2-alpha)

Twilight spotted the two-Warps confusion: a "Warps" row in the tile-meaning group **and** a "Warps"
row in Game View. They are different things — the tile-meaning group's one is a **tileset tile-trait**
(which tile graphics are warp-type, from `warp_tile_ids.asm`), the Game View one is the save's actual
**warp list** (the draggable objects). A warp is **map state**, not a tileset fact; and a **door** is a
passable **tile type** you walk across to reach a warp, which *is* a tileset fact. So:

- The **"Components" group is renamed "Tiles"** — it is the tileset's own tile meanings.
- The **warp tile-trait is removed** from that group; warps are shown by the Game View **object** layer.
- **Doors stay** as a tile trait.
- **Defaults (Twilight):** every **Game View** layer ON except the **Draw area** (so the player, people,
  warps, signs and screen box all show); every **Tiles** overlay OFF. To make warps visible without the
  old tile-trait, the Game View **object** Warps layer is now ON by default (it was off precisely because
  the trait covered it).

`tst_map_layers` updated + extended (the warp tile-trait must **not** be a Tiles row; the Game-View
defaults are pinned) — 12/12; `tst_map` 27/27.

### 2026-07-15 -- The "3-step wild encounter cooldown" checkbox, researched to the source (0.39.12-alpha)

Twilight briefed one checkbox for the **map details page**, and asked whether it was named/described right.
It wasn't quite: v2's `AreaPokemon::pauseMons3Steps` (save `0x29D8` bit 0) is `wStatusFlags2` bit 0 =
**`BIT_WILD_ENCOUNTER_COOLDOWN`** — the console's **post-battle cooldown**. The game **sets it after every
battle**, and on the next map entry (`EnterMap`, which the **Continue path reaches** via `SpecialEnterMap`)
a set bit loads `wNumberOfNoRandomBattleStepsLeft = 3` ("minimum number of steps between battles"),
decremented per step until it **self-clears**. So its save-editing effect is exact: **3 encounter-free
steps on the next Continue**. Console-verified **durable** — kept on load, so no yellow "!" — by a new probe
`scripts/emu/probe_wild_encounter_cooldown.py` (bit read back set + steps-left 3; the cleared control reads
0, proving the bit causes it and confirming the `$D13C` address). ⚠️ The clean `BaseSAV` already carries it
set (saved right after a battle). Renamed the model property `pauseMons3Steps` → **`wildEncounterCooldown`**
(+ doc naming the constant; byte-identical save output), exposed it on `MapModel` (a new `AreaPokemon*`
constructor arg, wired from `bridge.cpp`) with a setter that writes **exactly that one bit**, and placed a
`MapSwitch` on `DetailsPanel.qml`'s nothing-selected section labelled **"3-step wild encounter cooldown"**
with an honest one-line note. New note: `reference/wild-encounter-cooldown.md`. `tst_area_pokemon` 5/5,
`tst_qml_screens` 16/16, `tst_map` 27/27.

### 2026-07-14 -- Warps: the research pass, and the cartridge's verdict

Nothing in the app changed today. What changed is that we now **know what the warp bytes are** — and
several of them are not what the editor has been calling them for six years.

A warp is a door. The map's list of doors sits right there in the save, and the big question was whether
editing it does anything at all — because the game rebuilds that list from the cartridge on every map
load. **It does work**, and the reason is one line: as the game reads your save it deliberately flags
*"there is no previous map"*, which makes the very next map load **skip the rebuild entirely**. So the
doors you put there are the doors the console uses. The real Game Boy was asked, with a warp moved across
town and **a fourth door invented in a three-door town** — and it kept both. (It puts the map's original
doors back when the player walks out and back in, which the screen will say plainly.)

The rest of what the cartridge said:

- **Two of the flags can never survive a save.** *Scripted Warp* and *Is Dungeon Warp* live in a byte the
  game **zeroes every single time it loads a file** — along with four other flags elsewhere in the editor.
  Set them, save, load: gone. They'll be marked with a yellow **!**.
- **Two of the fields are dead.** *From Warp* and *From Map* are written by the game on every warp and
  then **never read by anything, anywhere**. A breadcrumb it drops and never picks up.
- **The two fields that actually matter weren't on the warps screen at all.** *"Outside is…"* — the town
  a building's exit puts you back in — and *"Wake up at…"* — where blacking out, Dig and an Escape Rope
  send you. Both were already in the save model, just filed somewhere nobody editing warps would look.
- **Two of them can hurt a real console.** The Fly destination has only **13** legal maps and the
  fall-down-a-hole destination only **12** legal map-and-hole pairs; anything else and the game reads
  whatever cartridge bytes happen to follow the table. The editor is **writing random illegal values into
  both today** whenever it places the player on a map — harmless only because that code path is asleep.
  It gets fixed before it wakes up.

Written up in full, with the console's own output beside it, and turned into a five-part plan for the map
screen: the doors become things you can **click, drag, add and delete on the map**, the toolbar grows
**actual tools** that make a warp / a person / a sign where you click, and every remaining byte gets a
name in English, its legal range, and — where it deserves one — a yellow **!**.

### 2026-07-13 -- The people walk the way the game walks (0.30.0-alpha)

Press **▶ Walk** now and you are not watching an impression of a Pokémon town. You are watching
`UpdateNPCSprite` — the game's own routine, transliterated out of the disassembly instruction by
instruction, running at the Game Boy's own 59.7275 frames a second. Bugs and all, because the bugs *are*
the behaviour.

Three things fell out of it that nobody would have guessed:

**Professor Oak is not standing still. He's turning.** A `STAY` sprite runs the entire
pick-a-random-direction routine — and the game writes the new facing *before* it checks whether the step
is legal. So Oak stands outside his lab choosing a direction, turning to face it, failing to move, and
setting a new delay. About once a second. Forever. That's what "STAY" means in this game, and our old
simulation — which simply skipped those sprites — was showing you a photograph.

**The game only animates the people near you.** The screen check is measured relative to the player, so
Pallet Town's fisherman, eight tiles below Red, doesn't take a single step until Red walks toward him.
That's the cartridge, not a shortcut.

**And the wander limit is broken — so ours is too.** Walk an NPC up five times and the game can never let
them walk back down. The disassembly says so itself, in a comment. We copy it, because a Pokémon town
where the NPCs *don't* drift into corners isn't a Pokémon town.

**Which means the "animation scratch" numbers aren't scratch.** The movement delay, the walk counter, the
frame counters, the step vectors — every one of them is the simulation's live state, ticking. They'll
have a proper home in the details panel, with a plain warning that the game rebuilds them all when the
map reloads.

**And a standing rule, from now on:** where the game has a file format, we use *the game's format*. Map
blocks already are. The music will be — it turns out we never needed a byte compiler at all, because the
game's music is *line-based assembly* with readable macro names, and a line parser reads it straight.

### 2026-07-13 -- Everything on the map screen actually works now (0.29.0-alpha)

The last round shipped a map screen where **the drag-and-drop didn't work, the characters didn't move,
you couldn't click a sprite, and deleting did nothing.** Three QML bugs were behind all of it, and every
one of them looked like a feature bug from the outside.

**Nothing redrew.** `brg.map.npcList()` is a function call, and a QML binding on a function call
evaluates *once* and never again. So the canvas kept drawing the cast it had memorised at startup —
which is why the walk simulation "didn't work" (it did; nothing repainted), why deletes vanished into
nothing, and why a dragged sprite snapped back.

**Drag-and-drop was never connected.** Qt decides what you're dropping onto by *geometric overlap with
the dragged item* — and the thing following the cursor was a decorative ghost while the real drag source
sat parked in the panel. It was never over the map, so nothing ever dropped. Same bug in reverse for
deleting: a sprite is clamped to the map, so it could never reach the panel you were dragging it to.

**You couldn't click a sprite.** Only click-*and-drag* would select one, the ✎ opened the panel in the
wrong mode, and the mouse could get stuck to a sprite. All three were the same thing: the drag handler
took the grab before the click ever completed. Sprites take a plain, unambiguous press/release now.

**And the player is a sprite too** — click him, drag him. There was never a reason he should be the one
thing on the map you couldn't pick up.

**No atlas.** The sprites are 72 loose PNGs you can open, look at and replace — `03_prof_oak.png` —
rather than one packed sheet nobody can edit. That should have been a question, not a decision.

**Music moved into the toolbar** as a chip, in the same language as the map picker beside it: a track
list, a sub-tracks toggle, the two save flags, and a ▶. The music *panel* is gone — a whole dock panel
for one combo box and two checkboxes was a panel too many.

**Tooltips only appear when you've turned tooltips on.** The map screen had been ignoring the header's
"?" toggle on purpose, and that was wrong — popping hints up on every hover is exactly the interrupting
clutter this app exists to avoid. One exception, and only one: the **!** on a character this map can't
draw properly. That's a notice, not a hint.

Also: the destructive-walk warning now **looks** like a warning instead of a page of blog text, the **!**
badges are centred, and the zoom slider and trackpad now go **straight through with no interpolation at
all** — only a mouse-wheel detent gets a 90ms bridge, and only because the hardware physically has no
data between one click and the next.

### 2026-07-13 -- The sprite editor was writing the OPPOSITE of what it said (0.25.1-alpha)

Four bugs, sitting in the sprite model since it was ported from the old editor in 2020. Every one of
them wrote a wrong byte into your save, and every one is now fixed and pinned by a test that fails by
name if it comes back.

**"Stay" meant walk.** The game has exactly two words for whether a person moves — `WALK` and `STAY` —
and we had their values **backwards**. So "this NPC stands still" wrote *wanders randomly*, for every
sprite, on every map. The real Game Boy settles it: boot Pallet Town and Professor Oak (who stands
still) reads `$FF`, while the Girl and the Fisherman (who wander) read `$FE`. We had it the other way
round.

**Every new sprite you placed was standing in tall grass.** There's a byte that tells the console to
draw grass *over* a character's legs so they look like they're standing in it. `$80` means "in grass".
We had that inverted too, so a blank sprite was born knee-deep in a field.

**Facing and movement were the same byte — and we were writing it into the wrong one.** The map data
says a person either has a wander *range* or a fixed *facing*, and those look like two different
things. They are **one byte**. We were taking the facing and writing it into the animation field, which
is a different byte in a different table — so the real value was lost, and the sprite quietly became a
free wanderer. Now one byte, one home, and a test walks all 900-odd sprites in the game to prove it.

**And five fields the game keeps, we never read at all** — a sprite's snapped-to-grid coordinates, the
direction it last bumped into something, the facing it had before it turned to talk to you, and the
game's own second copy of its picture id. They're all there now, and they all round-trip.

Also: every one of the 72 sprite pictures now carries a **group** — Story, Trainers, Townsfolk,
Pokémon, Objects — so the Characters bar has shelves to put them on.

None of this changes a save you don't edit. All 82 tests green, the cartridge included.

### 2026-07-13 -- The sprite set: what "cached sprites" actually was (0.25.0-alpha)

The old editor had a screen called **Cached sprites** — a dropdown of "Static List 1", "Static List 4
(Vermillion City)", "Dynamic List 9 (Route 5)" — and nobody could have told you what it was for.
Twilight's hunch was that the game doesn't even use it. **She was right, and the reason is worth
knowing.**

The Game Boy can only hold **eleven** overworld sprite pictures in memory at a time — nine that walk
and two that stand still. So every outdoor map names a **sprite set**, and every person, Pokéball and
boulder on that map has to be drawn from it. That's why the same faces keep turning up across a
region. There are ten of them, and the game names them properly: **Pallet & Viridian**, **Pewter &
Cerulean**, **Lavender**, **Vermilion**, **Celadon**, **Indigo**, **Saffron**, **Silence Bridge**,
**Cycling Road**, **Fuchsia**. (Twelve big routes are split down the middle and use one set on each
side of a line.)

Your save keeps a copy of the eleven pictures and the set's id — twelve bytes. **And the game throws
them away.** It wipes the id every single time it loads a map, including the load that happens when
you press CONTINUE, and rebuilds the whole thing from the map you're standing on. The bytes are a
scratchpad, nothing more. So the new **Sprite set** panel says exactly that at the top, in plain
words, instead of pretending an edit there will do something.

Everything else is there properly: the sets by their real names, what the game *would* load for the
map you're on (at the player's exact position — the split routes depend on it), a button to fill the
cache with it, and all eleven slots individually editable, with the last two marked as the still ones.

**And it caught a bug.** On a split route — Route 2, 5–8, 10–12, 15, 16, 18, 20 — the editor was
writing the *split* id into the save. The console never does that: it works out which side of the line
you're on **first**, and only ever stores one of the ten real sets. Fixed, and pinned by a test.

Written up in full: `notes/reference/sprite-sets.md`.

### 2026-07-12 -- The water moves (0.24.4-alpha)

The map has always been a photograph. On a real Game Boy it is not: the console quietly rewrites the
water tile every twenty frames, rolling it sideways, and on outdoor maps it cycles three flower tiles
on the twenty-first. About three times a second, forever. Our map drew frame zero and stopped.

It moves now — at the console's own pace, from the same single byte in your save that decides whether
anything animates at all. There is a play/pause and a step button in the status bar, and the pace is
shown next to them ("20f" or "21f") so you can see what the game is actually doing. A map that has
nothing to animate simply doesn't show the buttons.

**And writing it turned up two things we had been getting wrong for years.** The water was drawn as a
slide from 0 to 4 pixels and back; the console actually swings it from **−1 to +3**, because the
direction flips halfway through while the offset keeps accumulating. And the flowers ran in the wrong
order entirely — the game shows the first flower **twice as long** as the other two (1, 1, 2, 3), and
we ran 2, 3, 1, 1. Both were plausible. Both were invented. Both are now read straight out of the
game's own code and pinned by a test that checks each water step is a genuine *rotation* of the tile
rather than a second picture of it — because that is what the console does: it doesn't have a second
water tile, it rolls the one it has.

Hack values keep their promise too: an odd animation byte behaves as water-only, an even one as
water-and-flowers, exactly as the cartridge decides it — and setting it to zero kills the water and
**breaks Surf**, which the app will tell you.

(The fix also corrects the animated tiles in the name editor's previews, which used the same code.)

### 2026-07-12 -- The map gets a real layer system (0.24.3-alpha)

The chip bar is gone. In its place is a proper **layer tree**, in three groups, and it is the second
phase of the map overhaul.

**Guides** holds the block grid, a new **tile grid** (the 8-pixel tiles inside each block), the map
bounds and the border ring. **Meaning** holds the nine things you cannot see by looking at the art —
walls, grass, water, warps, doors, ledges, counters, cut trees, elevation edges. And **Game View**
holds what the Game Boy is drawing *right now*: the **player**, the **red screen box** (the 20×18
tiles he can actually see) and the **grey draw area** (the patch of map the game redraws around
him). Those last three used to be rectangles painted onto the map with no switch anywhere — you
could not turn off the red box sitting over your map. They are layers now.

Each layer has its own eye. Each **group** has an eye too, and it tells the truth: filled when
everything in it is on, half-filled when only some of it is, hollow when none — and one click
always changes something. Groups fold away when you are not using them. **Alt-click an eye** and
that layer goes solo — everything else in its group steps aside, and alt-clicking again puts back
exactly what you had, because a solo is a way of looking, not an edit of your setup.

Every row carries **the exact colour the map paints that layer in** — the colour comes *from* the
renderer — so the panel is the legend, and a legend that lives on the thing it explains cannot drift
out of sync with what is drawn. That is why the old footer legend is gone. A layer with nothing to
show on this map says so and refuses to switch on, instead of lighting an empty overlay and leaving
you wondering whether it works. And there is one dial: how hard the meaning overlays are painted,
because four shades of grey with three annotations stacked on them needs one.

**None of it touches the save.** A layer is how you look at a map, not a fact about it — and a test
now toggles every layer, every group, every solo, and demands the save come back **byte for byte
identical**.

### 2026-07-12 -- The map screen's foundations: two things that were quietly broken (0.24.1-alpha)

Before the map screen can let you edit the map, it has to be able to *reach* the map. It couldn't,
and finding out why turned up two faults that had been sitting there the whole time.

The first: the save's whole "area" — the map, the player, the tileset, the warps, the signs, the
people, the wild Pokémon, the flags — was **invisible to the interface**. Nine of its eleven parts
were marked in a way that makes Qt hand the screen an empty box instead of the data, without so
much as a warning. Two of them had already been rescued, one at a time, each after a feature
mysteriously did nothing (the music panel, this very week). Now all eleven are open, and a test
walks every one of them and every field inside, so a screen can never again be built on top of
nothing.

The second: **the warps, signs and people were not merely unshown — they were unreachable.** The
functions that hand one to the interface had never been registered with Qt, so asking for warp
number one didn't return an empty warp; it threw an error. Nobody had noticed, because until now no
screen had ever asked. They're registered now — and the test that guards this no longer keeps its
own private list of what's registered (which is how it stayed green while the app shipped broken);
it uses the app's own.

And one more, found while wiring it up: **the game's map database was never fully linked at
startup**, so every map's "what music plays here" pointed at nothing. That is why setting a map's
music has quietly been writing zero.

None of this shows on screen yet. It is the floor the new map screen stands on, and the floor was
not there.

### 2026-07-12 -- The Map screen, designed properly (notes only — no code yet)

The map screen is getting torn down and rebuilt, and this is the design it will be built from.

The research went into the three editors people actually use — **Tiled**, **Photoshop** and
**Aseprite** — and took the parts that earn their keep: a bar of options that **changes with the tool
you're holding**, panels that **collapse away to an icon** instead of piling up beside each other, a
properties panel that shows *the map's* fields until you select something and then shows *that
thing's*, **layers that live in groups** you can switch on and off together, and **objects** you can
just pick up and move. What it deliberately leaves behind is all of their chrome — this app is not
going to look like a 2003 desktop tool.

So: a **proper layer system**, in four groups — the guides, the meaning (walls, grass, ledges…), the
game's own view (**the player, and the red and grey boxes around him — layers now, at last**), and the
objects. Nothing stacks, nothing evicts anything, and everything you're not using folds away.
**Warps, signs and NPCs can be dragged around the map.** And **every single byte the save keeps for
the area** — around 120 fields, most of which have never had a screen at all — gets a home, an
explanation in plain English, and its full range, hack values and all, shown and never quietly
rewritten.

Two things were found while writing it down, and both are real. **The map's blocks live in the
cartridge, not the save** — so there is no honest way to "paint" a map in a save editor, and pretending
otherwise would have been a lie built into the UI. And **the map has never actually moved**: the game
re-writes the water tile *every twenty frames*, rolling its bytes sideways, and cycles three flower
tiles on the twenty-first — we have been drawing frame zero, forever. The water is dead and the
console's water is not. That is now a bug with a fix, not a feature request.

The plan is thirteen phases, each one finished before the next one starts.

### 2026-07-12 -- The map tells you what it MEANS (0.24.0-alpha)

Until now the map screen could draw Pallet Town. It could not tell you a single thing about it —
which squares are walls, which grass will throw a Pokémon at you, which tile is the door, which
ledge you can only jump one way. None of that is visible by looking at the art: a wall and a floor
are just two pictures.

Now it is. A row of **Show** chips over the map turns on **Walls, Grass, Water, Warps, Doors,
Ledges, Counters, Elevation, Cut trees** and the **border ring** — each one a soft tint and its own
little pattern, laid over the map and faded in, so several can be on at once and still be read
apart. They are **off by default**, because the map is the point. And a chip for something this map
hasn't got says so instead of switching on nothing (Pallet Town has no ledges, so the Ledges chip
is politely greyed out).

Turn on Grass in Pallet Town and you can see **Route 1's tall grass bleeding into the border ring**,
which is exactly what the Game Boy does.

**Click any block** and a Blocks panel opens with it drawn big — all 16 of its tiles, each saying
what it does in words ("Grass — wild Pokémon", "Ledge — jump down", "Counter — talk across it").
Hover a tile there and it lights up on the map.

**The "Outdoor" toggle was lying.** There are three of these, not two, and the third one is Cave.
The button is now a proper **Indoor / Cave / Outdoor** control that says what each one *does* —
nothing animates, water animates, water and flowers animate. That's a real fix, not a relabel:
every cave in the game has been rendering with **dead, motionless water** because "Cave" was being
lumped in with "Indoor".

There is a new **Tileset panel** for everything the tileset owns — and **not one field in it is a
number you type**. The grass tile and the three counter tiles are picked by *looking at the actual
tiles*. The out-of-bounds block is picked by *looking at the actual blocks*. Even the two leftover
bytes from the last Strength push — which the game rewrites the moment you shove another boulder —
get honest editors instead of being hidden, because every byte in the save is yours.

#### 🐞 And it found a bug that was corrupting saves

The tileset data had **three tilesets pointing at the wrong collision list** — Mart, Forest and
Red's House 2. So asking the editor to put you in a **Poké Mart** wrote a **bedroom's** walls into
your save: wrong walls, wrong door, wrong counter, in the real game.

The cause is a lovely trap: several tilesets *share* one list in the cartridge, and whoever built
the original data assumed one list each, in order — so the whole chain slipped by one. Verified
against Twilight's own cartridge (all 24 collision lists re-read straight out of the ROM and
matched byte-for-byte), fixed, and pinned by a test that re-derives every pointer from the data —
put the bug back and it fails by name.

### 2026-07-12 -- All 151 pieces of music (0.23.0-alpha)

The track list is now **everything the game has**: the 46 songs, and the **105 "inner voices"** hiding

### 2026-07-12 -- ...and the actual Game Boy agrees, note for note (0.22.1-alpha)

The music isn't just playing now -- it's **provably the same music the cartridge plays**.

The game's sound engine keeps everything it knows in 243 bytes of memory. So the test suite boots the
real cartridge, plays a track on it, and photographs those 243 bytes **on every single frame** -- then
hands the first photograph to our engine and demands it produce all the others. Every note, every delay,
every wobble of vibrato, in step.

**All 46 tracks pass.** So does a "glitch" track, for good measure.

It caught three real mistakes on the way in -- including a wonderfully odd one: the game's pitch-slide
routine quietly scribbles over the note's own pitch, so a sliding note *starts on the wrong frequency*.
That's not a bug in the game so much as a fingerprint of it, and now we have it too.

### 2026-07-12 -- The editor plays the game's music (0.22.0-alpha)

The Map screen has a **♪ Music** panel now, and it is not a list of filenames: the editor has **a Game
Boy sound chip in it**, and the game's own sequencer running on top.

There are no audio files anywhere. `pse-audio` is a model of the DMG's four channels -- two squares, the
wave table and the noise generator -- and `Gen1SoundEngine` is Pokémon Red's own music code, ported line
for line, reading the tracks the game reads and poking the same registers 60 times a second. What comes
out of your speakers is what the cartridge does, arrived at the same way.

So the panel gives you: the map's **music track** (the map's own is selected the moment it opens), the
two save flags it has never shown before -- **No Audio Fadeout** and **Prevent Music Change**, each
explained by what it *actually does* rather than what its name implies -- and a **▶**.

And **hover-preview**: with the music playing, run the mouse down the track list and it changes as you
go, so you can skim the whole soundtrack. Hovering auditions; **clicking commits**. Moving a mouse never
touches your save, a line tells you plainly when what you're hearing isn't what's stored, and letting go
snaps you back to the truth. Audio never starts on its own, and closing the panel stops it -- nothing
hums away behind another screen.

Also fixed, quietly: setting a map's default music used to write the **bank** number into the **track
id**, which is a different song entirely.

### 2026-07-12 -- Even the glitch maps render, and nothing was invented to do it (0.17.1-alpha)

The unused and half-finished map ids -- the ones with no size and no tileset, that the editor had been

They were never blank. They are unfinished **copies**, and the data said so all along: every one of those

### 2026-07-12 -- Red is standing on his own doorstep (0.21.0-alpha)

The player is on the map. He is facing the way the save says he is facing, and he is standing four pixels
higher than you would think -- because that is what the Game Boy does, to make a character look like he is
*in* a tile rather than sitting on top of one. That is not a number anyone guessed: the console's own object
memory was asked where it had put him, and it said (64, 60), which is precisely where we had already worked
out he should be from the other direction entirely.

Three things about Gen 1's sprites are quietly delightful, and each one is a trap for anybody who assumes.
**There is no right-facing Red.** Facing right is facing *left*, mirrored. **There is no second walking
frame** either -- the "other foot" is the same drawing, flipped. And the bottom half of every character
carries a priority flag, so that tall grass can be drawn *over his legs* while staying behind his head.

And there is a payoff that has been sitting in wait since yesterday. Two of the six glitch palettes --
contrast 1 and 2 -- looked completely harmless: the map came out perfectly normal. That was never because
they were harmless. It was because their damage is in the *object* palette, and there were no objects. There
is one now. Set the contrast to 1 and the world looks fine and **Red does not**, which is exactly, precisely
what the cartridge does.

### 2026-07-12 -- The six glitch palettes are real, and now they render (0.20.0-alpha)

The save has a byte the editor has always called "contrast", and it turns out not to be a brightness dial at
all. The game takes that number and **subtracts it from a pointer** -- a pointer into its little table of
fade palettes -- and then reads three bytes from wherever it lands.

Land on a multiple of three and you get a real palette: normal, dark, the near-black one caves use when you
haven't got FLASH, and black. **Four levels.** Land anywhere else and you read *across the seam between two
of them*, and out comes three bytes that were never a palette at all. **Six of those.** Four and six, which
is exactly what Twilight said before anyone went looking.

So the map is now drawn through whichever palette that byte actually produces -- and that means the glitch
palettes render as the genuine article rather than an impression of one. We are not simulating "broken"; we
are applying the byte the console would be holding. All ten were checked against the real hardware's palette
registers, and all ten match.

There is a lovely wrinkle. Contrast 1 and 2 look perfectly normal on the map, because the damage is in the
*sprite* palettes rather than the background one. They are lying in wait, and they will show themselves the
moment the player is drawn.

Also: the map can now be pinched, dragged and scrolled in both directions, and zoom lands on whatever you
were pointing at rather than the far corner -- which it needs to, because the maps are only getting bigger.

### 2026-07-12 -- The border isn't a wall of trees. It's the next route. (0.19.0-alpha)

A Pokémon map does not stop at its edge. Stand in Pallet Town and look north and you can already see Route
1 -- not a picture of it, *it*: the game keeps every map ringed by three blocks of its neighbours, bled in
from their real block data, which is why walking out of a town is seamless rather than a cut. Get that
wrong and the world stops feeling like a world.

It is also, by some distance, the nastiest thing in the map engine. One signed offset gets clamped into two
different numbers -- one saying where the strip lands in *your* ring, the other how far into the *neighbour*
it starts. The length can be cut short by either map. The very same length byte means a *width* going north
and a *row count* going east. And the source is not an index into a tidy array at all: it is a raw pointer
into another map's data in another ROM bank, which the cartridge will happily read straight off the end of.

So it was not written from reasoning. It was written from the cartridge, and then checked against it: **all
78 connections in the game, every byte of every one, identical.** The ring it builds is byte-for-byte the
ring the Game Boy builds.

Two things fell out along the way. The editor's old connection formula turns out to be **wrong** -- it
branches on a test the real game never makes -- and the mysterious `flag` in the map data is a patch for
exactly that. Neither is used any more; the real one needs no flag. And blocks are now addressed the way the
console addresses them -- bank and pointer -- so if a hand-edited connection ever runs off the end of its
neighbour, it will run off into precisely what the console would have found there, rather than into
something merely plausible.

### 2026-07-12 -- The Game Boy now checks our homework (0.18.0-alpha)

Up to now, every test this project has asked was some version of *"is the editor consistent with itself?"*.
That is a fair question, and it is not the real one. The real one is **"does the actual game agree?"** — and
we can finally ask it.

`tst_emu_parity` boots the genuine ROM in an emulator with one of our save files, waits for the game to load
it, and then reads **the console's own RAM** back out. Not our idea of the map: the map the Game Boy built,
in the memory it built it in. Then it demands we match it byte for byte.

We do. The pointer the game computed for where its view starts, the blocks it assembled the map from, the
24x20 scratch area it expanded them into, and the **20x18 tiles it was actually displaying** — all identical
to ours. That last one is the whole view pipeline in a single array, with no sprites and no palettes in the
way, and it means the map emulator is right.

**With one exception, which it found on its first honest run.** Our border ring is a wall of the map's border
block. The real one isn't: the game bleeds the *connected maps* into it, so Pallet Town is ringed by the
bottom of Route 1 and the top of Route 21. We had that on the roadmap; now the console has put a date on it,
and the test is written and waiting -- deliberately as an expected failure, so the day connections land it
starts passing and refuses to be forgotten.

The ROM is Twilight's own cartridge backup. It is git-ignored, has never been committed, is never shipped,
and everything that touches it degrades to a polite skip without it -- so none of this is anything anyone
else has to have, or is ever handed.

### 2026-07-12 -- The map, actually emulated: the game's own buffer, drawn (0.17.0-alpha)

**The old Maps screen is gone** — a greyed-out tile leading to a menu of dead ends — and in its place is
a map screen that doesn't *depict* the map so much as **rebuild it the way the Game Boy does**.

The game never holds a map on its own. It holds it **ringed by a 3-block border**, fills that ring with
the map's border block, and draws its screen out of a **6×5-block scratch area** that slides a **20×18-tile
window** around inside itself in half-block steps. That is the whole trick behind Gen 1's smooth scrolling:
the picture moves while the grid it is built from never does. All three of those boxes are now on screen,
in their real places, over a map rendered block-by-block, tile-by-tile, at one screen pixel per Game Boy
pixel — from the `.blk` and `.bst` data imported **verbatim** out of the pret/pokered disassembly.

**And it can prove it.** A real save carries the pointer the Game Boy itself computed for where its view
starts. We recompute that pointer from nothing but the player's coordinates and the map's width — and it
comes out **identical, byte for byte, on both real saves**. That single test is the difference between an
emulator and an impression, and it's the one that fails first if our understanding ever drifts.

Along the way, two things the ROM does that no one would design on purpose are now **reproduced rather
than tidied up**: one map declares 96 blocks but only stores 92, so the game reads its last row out of
the *next* map's data — and we read it the same way. Twenty-two glitch map ids that our data calls empty
turn out to quietly point at real maps' headers in ROM. That second one is left alone, on purpose: it's a
data question, and the data is Twilight's.

Everything the screen knows is pinned by 18 new tests. The player sprite, the connected-map strips that
bleed into that border ring, and the tile animations are the next steps.

### 2026-07-11 -- Keyboard: a real modifier drops the latched page (0.16.6-alpha)

**The bug Twilight found, and it was a real one.** Latched and held modifiers were OR'd together, so:
click "Uppercase" in the page strip to poke around, then start typing — and **Shift does nothing**,

**Touching a physical modifier now drops the latch.** The moment you press a real Shift/Ctrl/Alt you're

And: the **tiles no longer overflow their keys** (the glyph — and the white card behind the picture tiles
— is sized to fit *inside* the cap, with the cap clipping as a backstop), and the footer preview is

### 2026-07-11 -- Keyboard: the box you can type, no fake key combos, and two modes that behave (0.16.4-alpha)

**Type a border.** Twilight's idea, and a better one than mine: the box-frame glyphs are laid out **as
the box** on the Tiles page —

```
    Q  W  E        ╔  ═  ╗
    A  S  D   -->  ║     ║     (S is the box's hollow middle, and is empty)
    Z  X  C        ╚  ═  ╝
```

— so a border is literally `Q W W W E` … `Z X X X C`. The keys *are* the picture. This is why the
horizontal and vertical edges are the **only two duplicated tiles in the whole map** (you can't draw a
box with one vertical edge), and `tst_font_keyboard` now pins exactly that: every tile reachable, and
those two the only ones on more than one key. The cursor arrows moved to `, . /` — arrow-ish keys, and
it keeps the "things that point" family together.

**No key may pretend.** The rule, stated: *a tile goes where a real keyboard would put it whenever it
can, and must never fake it.* An alternate period on **Ctrl+/** — while a perfectly good period sits on
the period key one layer down — is a false affordance, so it moved to a spare key. And when the deck
types, on this key with these modifiers, exactly what the physical keyboard would (`a` on A, `!` on
Shift+1, `?` on Shift+/), **the corner legend disappears**: there is nothing left to teach, because the
key is already telling you by being that key. It stays on every Ctrl/Alt key, where a real keyboard
types nothing and the legend is the only clue you have.

**Edit mode live-updates.** Everything you type in the text field now reaches the name immediately, so
the preview keeps up with you — and **discard still discards** (the value from when you entered edit
mode is remembered, and the cross puts it back). In keyboard mode the field is grey, not black, with a
**soft caret pulsing at the end of the text** — the invitation to type, sitting exactly where the next
key will land.

**Name/Example can no longer resize the keyboard.** The footer was sized to its preview, so flipping
that toggle grew the footer, which shrank the body, which shrank the **deck** — the whole keyboard
re-flowed under your hands because you asked to see an example sentence. The footer is a fixed height
now, and its content hangs *up from the bottom* so the byte counter sits at the edge in both modes.

Also: the **"Simulated" controls moved out of the top bar** (where they floated over a name field they
have nothing to do with) to a tray **directly above the keyboard, on the tile pages only** — next to the
only thing they affect. A **quiet page description** sits under the deck. The key detail pane **scrolls**
long descriptions instead of eliding them, and fits the previews of the codes that expand to **fifteen
characters** instead of slicing them off. Every button in the name row has the same chrome (some had a
background and some didn't). And the idle text was simply **wrong** — a key is not "one game character",
it's **one byte**, which might print one character, several, a whole name, or nothing at all while it
changes the text.

### 2026-07-11 -- Keyboard: punctuation lives on its own keys now (36 keys -> 47) (0.16.3-alpha)

Twilight: *"Have some of the symbols like '?', '.', ',' amongst others actually be on their
corresponding key."* Right — and that isn't a tweak, it's the map. The punctuation keys
(`` ` `` `-` `=` `[` `]` `\` `;` `'` `,` `.` `/`) were **dead filler**, which meant `.` and `,` were
exiled onto a number row they have no business being on. They're **real keys** now, so the deck went
from 36 assignable keys to **47** and the whole tile→key map was re-authored around them:

- **Base layer:** `.` types `.`, `,` types `,`, `/` types `/`, `;` types `;`, `'` types `'`, `-` types
  `-`, `[` and `]` type `[` and `]`.
- **Shift layer:** the real shifted keys, wherever the game has that glyph — **Shift+/ = ?**,
  **Shift+; = :**, **Shift+' = "**, plus the number row's **! $ × ( )** (`×` on the `*` key, because
  `*` is multiply). The glyphs the game simply doesn't have (`@ # % ^ & _ + { } | < >`) leave their
  keys empty; the spare number-row slots take the Normal characters with nowhere else to be (♂ ♀ Pk Mn).
- **Ctrl layer:** the punctuation that has a real partner one layer up — the closing `"` under the

### 2026-07-11 -- Keyboard: Caps locks the Shift PAGE, and the whole screen got its look fixed (0.16.2-alpha)

**Caps Lock is a page now, not a letter-case rule.** The previous cut gave it the real-keyboard
behaviour (letters only; the number row kept typing digits) — and Twilight immediately hit the problem
that creates: *"caps lock and holding down shift are 2 ways to get page 2, so why are there different
page 2 depending on caps lock, i think something got crossed."* Nothing was crossed, but the model was
wrong: caps-on-letters-only produces a layer that **isn't one of the 8 pages**, so the page strip could
not say where you were. A model the UI cannot display is a bad model however "correct" it is. Caps now
locks the **Shift page** entire — Shift inverts it, Ctrl/Alt ignore it (Ctrl+B is still bold B) — so
every state the deck can be in is exactly one page and the strip is always right. The trade: the
punctuation row rides along with caps, so typing a digit means tapping caps off. Rare (digits aren't
in-game-legal in a name), and the deck *shows* the row change.

**The look.** Twilight: *"colour was supposed to be visible at a glance, not only on mouseover"*,
*"keyboard looks really bad but i cant place my finger on why"*, *"the colours on the top and bottom
are really ugly and the top and bottom bar is way too tall"*. All fair, and all diagnosable:

- **No figure/ground.** Light-grey caps on a light-grey chassis on a light-grey pane. The deck now has
  a **dark slate body** with light caps, like a real keyboard — and the structural keys (Caps/Shift/
  Ctrl/Alt/Enter/⌫) are **dark** with light text, so the eye instantly separates "keys that type
  something" from "keys that do something". A held or latched modifier goes **bright**.
- **The category colour did nothing.** It was a 6% wash behind a 35% border — literally only visible on
  hover, which defeats the entire point of having a colour legend. Caps now carry a **real wash** of
  their category colour with a solid border. On the dark body the deck reads as coloured regions the
  moment you look at it.
- **The bars.** The washed-out `lighter(accent, 1.5)` blue stripes are gone: both are now a clean light
  surface with a hairline divider. The header went **132 → 88px** and the footer **119 → ~104px**, and
  the keyboard got the difference — it's the point of the page and it was being squeezed into whatever
  the bars left over.
- **The spacebar looked like a dead key**, because it was rendering the Space *tile*, and the Space tile
  is — correctly — blank. It says **"Space"** across it now. It is, and always was, Space on every page.
- **You couldn't read the key legend** — the small letter in the corner that tells you which key types
  this tile, i.e. the thing that makes "just type it" discoverable at all. It was 9px of mid-grey at 75%
  on a near-white cap. It now scales with the key, sits at full opacity, and is drawn in a dark shade of
  the cap's own category colour.
- **The deck didn't look like a keyboard.** It was a floating block of 36 caps — roomier than a real
  layout, and somehow worse. The shape you recognise as "a keyboard" comes from the ragged edges as much
  as from the letters, so the deck now draws the **full ANSI silhouette**: `` ` ``, `-`, `=`, Tab,
  `[ ]`, `\`, `;`, `'`, `,`, `.`, `/` and Win/Menu are all there as **dead keys** — muted, inert,
  unclickable, purely there so the thing reads as the object it's imitating. The real keys give up ~10%
  of their size for it; the deck scales, so a wider window hands that straight back.

### 2026-07-11 -- Keyboard: lowercase is the base layer, Caps Lock is real, and the field has two modes (0.16.1-alpha)

Twilight, on the first cut: *"its counterintuitive to have shift lowercase... a-z should be without
shift and uppercase with."* She's right, and it's the one place the deck contradicted every keyboard
its user has ever touched. (My reasoning had been that Gen 1 names are all-caps — but that's an
argument for a good Caps Lock, not for inverting the alphabet.)

So: **the base layer is now lowercase** (`a–z` + `0–9`) and **Shift is uppercase** (`A–Z` + the
punctuation row) — exactly like the keyboard under your hands. The number row didn't move: digits
unshifted, symbols shifted. No other page changed, and every tile still appears exactly once.

**Caps Lock is a real Caps Lock, not a latched Shift.** It affects the **letters only**, so the number
row keeps typing numbers and you can type `PIKA2` without unlocking; it's **ignored under Ctrl/Alt**
(Ctrl+B is bold B whether or not the light is on); and **Shift inverts it** (caps + Shift = lowercase).
That's three rules the user already knows, so the rules live in C++ (`FontKeyboard::pageForKey`) where
`tst_font_keyboard` pins each one. It also means the deck can honestly show two pages at once —
uppercase letters above a digit row — which is what a keyboard does. Qt gives no portable way to *read*
the caps light, so if it was already on before the screen opened the deck corrects itself from the
OS's own answer (`event.text`) on the first letter you press.

**Physical modifiers are momentary; clicking latches.** Hold Ctrl and the deck flips to that page; let
go and it drops straight back — nothing sticks. Clicking an on-screen modifier cap (or a page button)
still latches, because a mouse can't hold a chord and click a key at once, and because a latched page
is the only way in when the OS eats the chord.

**The name field now has two explicit modes**, and the screen says which one you're in:

- **Keyboard mode** (the default): the field is a read-only display of what the deck is building, and
  Backspace removes a whole **tile**.
- **Edit mode**: the pen button turns it into an ordinary text field — caret, selection, Ctrl+C/V/Z,
  and a character-by-character Backspace, as a text field should behave. **The whole keyboard fades out
  and goes dead**, because it has no say in what you're typing and shouldn't pretend otherwise. The pen
  is replaced by a **check** (apply) and a **cross** (discard) — so an edit is something you commit or
  throw away, never something that half-happened. Nothing you type in edit mode reaches the name until
  you press the check.

### 2026-07-11 -- The full keyboard is an actual keyboard now (0.16.0-alpha)

The full-screen name editor has been rebuilt from scratch. It used to be a filter form next to a
wrapping list of coloured chips — a thing you had to *search*. It is now a **keyboard**: an
ASDF/QWERTY deck whose 36 letter and number keys each carry one game tile, with the physical key that
types it printed in the corner of every cap. Click a key, or just type it.

**255 tiles over 36 keys needs 8 pages — and Shift/Ctrl/Alt give exactly 8 combinations.** One page
per chord, nothing left over. You reach a page by *holding* the modifiers (like a real shift layer),
by *clicking* the on-screen modifier caps (they latch, so mouse and touch users never hold a chord —
and so a page stays reachable when Windows eats the combo: Shift+Alt and Ctrl+Shift switch keyboard
layout on multi-language setups, and Ctrl+Alt is AltGr), or by clicking its button in the page strip,
which shows the chord so it teaches the shortcut instead of hiding it.

The map is **not** a linear dump of tiles into slots. Three rules, in order: **identity beats
everything** (page 1 is A–Z + 0–9 and page 2 is a–z, exactly where a keyboard puts them, so there is
nothing to learn); **mnemonic beats ergonomics** (`'s` on **S**, bold **B** on **B** — Ctrl+B, the
same thing it means in every other program — `<player>` on **P** with `<rival>` on **O** beside it,
`<pc>` on **C**, `<dex>` on **X**); and **ergonomics breaks ties** (the more-used tile takes the
better key — home row, then upper, then bottom, then the number row). Pages run in the category
order, so the four *cheapest* chords hold every Normal, Single-Char, Multi-Char and Variable tile —
i.e. everything you'd ever legitimately put in a name — while the Pictures and the Control codes that
*glitch* a name sit behind two- and three-key chords. You cannot hit `<end>` by accident. The six
box-frame glyphs are laid out **as a box on the keys** (Q ╔, W ═, E ╗ / A ║ / Z ╚, X ╝) and the three
cursor arrows take the right home keys. The **Space tile rides the real spacebar** — the one tile not
on a letter or a number, because that is where every human already expects it.

Every one of the 255 tiles appears **exactly once** across the 8 pages. That isn't a claim in a
comment — `tst_font_keyboard` (39 cases) fails if any tile is duplicated or, far worse, silently
stranded with no key at all.

Also: keys animate their tiles with the tileset (one shared sheet for the whole deck, so 36 animated
tiles cost one render per frame rather than 36 tileset rebuilds); Backspace deletes a whole `<code>`
rather than biting one character out of the middle of one; a key that won't fit shakes the name field
instead of failing silently; hovering any key fills the detail split with its render, name, code,
category and description; and the old category *filter* on the left is now a colour *legend* (there's
nothing left to filter — the deck shows everything). The name box stays a real text field: click into
it and Ctrl+C/V/Z work as normal, and the deck dims its key legends to show it has handed over the
keys.

Gone with it: the character-chip list, the filter sidebar, the raw tilemap view and its Grid/Tileset
toggle, and the `FontSearchModel` behind them.

### 2026-07-11 -- Exchange prices the WHOLE trade, not each step (no more phantom refunds)

Fixed the exchange economics. The trade was priced **per step** — each item you asked for was
separately rounded up to a whole number of the item you were paying with, and each one handed back
its own leftover as money. That massively overcharged and over-refunded: asking for 3 Fresh Water
(₽200 each) took **3** Potions (₽300 each) and refunded ₽300, when 3 Fresh Water is ₽600, which is
**exactly 2 Potions and no change at all**.

Now the whole trade is priced **once**: the total value being bought is rounded up to a whole number
of the given item, and only that single leftover comes back as money. A trade that divides evenly
costs nothing extra and **refunds nothing** — and the rate line under the button drops the "+₽0"
entirely rather than showing a refund that isn't happening. Verified live: 3 Awakening (₽600) from
Antidotes (₽100) now reads exactly `-6 Antidote` with no refund and money unchanged; 3 Fresh Water
from Super Potions (₽700) takes 1 Super Potion + ₽100 back instead of the old 3 + ₽1,500.

The rate lines also now describe the **queued** trade rather than a fictional per-unit rate, so what's
written on Checkout is literally what the card shows. `perAGive`/`perARefund`/`perBGive`/`perBRefund`
are replaced by `giveFor(dir, steps)` / `refundFor(dir, steps)`, which the preview, the "+" gating,
and `checkout()` all share — one pricing path, so they cannot disagree. Money still only ever goes up.
`ctest` green (72/72).

### 2026-07-09 -- Resolved the type-2 "single truth": single types write duplicate-of-type1

Settled the last tracked type-2 decision straight from the pokered disassembly. A single-type
Pokémon is stored as a **duplicate of type 1** (`data/pokemon/base_stats/*.asm` — e.g. Charmander is
`db FIRE, FIRE`), `0xFF` is not a valid type at all (`constants/type_constants.asm` runs `$00..$1A`),
and `engine/pokemon/add_mon.asm` copies both type bytes verbatim from the base-stats header into
party/box data. So the game's — and now the editor's — canonical written form for a single type is
duplicate-of-type1.

`save()` already emitted that for a non-explicit single type, so on-disk output is unchanged for
normal saves; **byte fidelity is untouched** — a save that literally stored `0xFF` (only ever a
hacked/glitch save) still round-trips as `0xFF` via `type2Explicit`. The real fix: every editor
(re)generation path — `randomize()`, `correctTypes()`, and `update(resetType)` — now clears
`type2Explicit`, so a *generated or corrected* single type serialises as the duplicate rather than a
stale `0xFF`. `isCorrected()`'s tolerance (accept either `0xFF` or the duplicate as a valid single
type) is now the finalized, intended behaviour — the "temporary dirty patch" framing is gone. New
regression test `box_singleTypeCanonicalForm` pins it; the load/save/isCorrected comments and
`reference/gen1-knowledge.md` now document the resolved truth.
