**One recorded divergence — CI owns the release tag.** The hub runbook ends a release with a manual
`git tag`, but this repo's `release.yml` already derives `v<VERSION>` and creates the tag itself
(softprops) on the `main` push, and its `version` job *skips the release if the tag already exists*.
Following the runbook literally would make every release silently no-op. So we diverge on purpose:
the `--no-ff` merge to `main` is the release act and **CI applies the tag** — do **not** `git tag`
by hand. Documented in `git-workflow.md`, `versioning.md`, and the `release.yml` header comment
(comment-only; the pipeline is functionally unchanged → no CI risk).

here now ends with an honest report the hub reads to improve the standards.

the compliance audit + process-reports model in `cross-project-sync.md`.
