## s16j — Indoors there is no sprite set — `0.32.0-alpha`

Twilight's fourth pass, and the one that sent me back to the cartridge:

> *"The sprites need to render as accurately as possible if dropped… If changing the value in the save
> file won't make a difference because it's overwritten on load, then the map emulator should emulate
> that. **So no matter what sprite set is set to, the map should render exactly and completely
> accurately, like the game would after loading a modified save file.** Please research and figure out
> how to properly and accurately render the sprites dropped which aren't on the sprite list — **and
> does the sprite list change if the NPCs are different?** … If it is recalculated, those yellow
> exclamation points probably need to change."*

They did. I read `engine/overworld/map_sprites.asm` end to end and **we were showing the wrong
warning**.

### The answer, and it is a big one

**Indoors there is no sprite set.**

```asm
InitMapSprites::
    call InitOutsideMapSprites
    ret c                              ; outdoor maps are already done
; if the map is an inside map (i.e. mapID >= FIRST_INDOOR_MAP)
    ld a, [hl]                         ; ⚠️ the NPC's OWN picture
    ld [de], a
    ...                                ; falls into LoadMapSpriteTilePatterns
```

Indoors the game loads **each character's own artwork**. The cast *is* the set. Anybody can go
anywhere in a building — until the **video memory** runs out, and that limit is real and small:
**10** walking slots and **2** four-tile still slots.

**Outdoors** the set comes from a **ROM table** (`MapSpriteSets[wCurMap]`, splits resolved against the
player), and the NPC→VRAM lookup is a loop with **no bounds check**:

```asm
.getPictureIndexLoop
    inc c
    ld a, [de]
    inc de
    cp b
    jr nz, .getPictureIndexLoop   ; ⚠️ runs off the end of wSpriteSet, into WRAM
```

A picture that isn't in the set walks off the end of the table until some byte happens to match. So
"the game would draw it as garbage" is *literally* true, and the garbage depends on the contents of
RAM.

**We were asking the save's cached sprite set** — the twelve bytes the game throws away. On an indoor
map that meant flagging characters that would have drawn perfectly; on any map, editing the cache
silently changed our answer. `MapModel::vramPictures()` is the console's routine now, and everything
that asks "will this render?" goes through it. Three tests pin it.

### The walk finally looks like a walk

Two bugs, and the second one was the frame rate.

**They teleported** because `TryWalking` moves `mapX`/`mapY` to the **destination** at once and *then*
slides the sprite a pixel a frame for sixteen frames. The tile is where they are *going*. The canvas
gets the sub-tile offset now, and the legs move: `SpriteFacingAndAnimationTable`, verbatim — which is
where I learned there **is no second walking frame**. Frames 1 and 3 are the same picture, and 3 is
mirrored. *That mirror is the other leg.*

**The frame rate collapsed** because the cast moving emitted `MapModel::changed()` → `sourceChanged()`
→ **the entire map image was re-rendered**. The simulation moves somebody ~60 times a second. It was
re-rendering the whole of Pallet Town sixty times a second to shift one 16×16 sprite. A sprite moving
does not change one pixel of the map: it has its own signal now (`castChanged()`).

### And the rest of her list

* A **"Reloaded values"** switch, hard right on the toolbar, **off by default** — a third of a sprite
  is bytes the console recomputes on load, and with the switch off they are **not built at all**.
* The picture picker dropped you back to the map's details the moment you opened it: its **modal**
  overlay leaked the dismiss-press onto the canvas, whose ground-tap clears the selection. The picker
  is no longer modal, and `MapCanvas.popupsOpen` makes the leak structurally impossible.
* **The contrast strip wears the real palette now.** Each segment is painted in the four shades that
  value actually renders in (the genuine `rBGP` byte, glitch reads across the fade seam included), so
  sliding along it shows you the map going dark before the map does.
* Enum options come **sectioned** — the ordinary values, then the flagged ones under their own
  heading. A **!** on row 84 of a flat list of 140 is a **!** nobody sees.
* The script picker lists **only the scripts this map has**. It used to offer all 64, with fifty rows
  of *"this map has no script 37"* burying the seven that mean something.
* *"Something else…"* is **sticky** — it no longer slams shut because you typed a value the combo has
  a name for.
* Layers: **Game View at the top**, and the defaults are hers: block grid, map bounds, doors, warps,
  player, people & objects, screen box. The screen box and draw area **follow the player as you drag
  him**.
* Music: **"No music" at the top**, sub-tracks **off**, and the combo delegate no longer draws a group
  heading above *every single row* (it was reading `group`, which every row carries, instead of
  `header`, which only the first of each group does).
* **The Walk chip** was pale text on a near-white chip. Both states are spelled out now, not inherited
  from a palette that resolves light. The screenshot review walked past this twice; it is written up
  in `ui-patterns.md` along with the scrollbar lane, as the two things that keep coming back.

## s16i — The panel that was a wall of numbers — `0.31.0-alpha`

Twilight's third pass, and the sharpest:

> *"The sprite details page looks bad. The Who When Where is really really dumb, don't do it. Second,
> the fields are all just raw values — exactly what I said not to do… I don't know what most of those
> numbers mean on sprite details, it's cryptic as crap… **Don't show boxes if it's unnecessary** —
> some of these raw values have a combo box next to them, and I bet that combo box value is going to
> determine if the textbox raw value is even needed to be there or not."*

That last sentence is the whole design, and she found it by looking at the screen. The panel now works
the way she described.

### Every field declares a KIND, and the kind picks the control

`MapModel::npcFields` used to emit one row per byte: a number box, a paragraph, a heading called *Who*.
It now emits a **kind**, and `SpriteField.qml` draws what that kind deserves:

| kind | the control |
|---|---|
| `picture` | a grid of the actual **artwork** — you pick a character by looking at them (`PicturePick.qml`) |
| `coords` | X and Y **together**, in one control. They are one fact |
| `enum` | a combo — and **the raw byte box only appears when the combo cannot say the value** |
| `frames` | a **duration**, drawn as one: a bar, the frame count, and the seconds at 59.7275 Hz |
| `team` / `byte` | the last resorts |

Nothing is refused and nothing is corrected. The full byte range is always one click away, behind
*"Something else…"* — it just isn't sitting on the screen when it has nothing to say.

### The save's own bits decide what the panel SHOWS

The text byte is three facts in one (`constants/map_object_constants.asm`):

```asm
const_def 6
const BIT_TRAINER  ; 6
const BIT_ITEM     ; 7
```

So the top two bits say what a sprite **is**, and the bottom six are its script id. The panel reads
them: an **item ball** gets an item picker and **no trainer roster**; a **trainer** gets a class and a
roster and **no item**. Setting one half never rewrites the other — byte fidelity, applied *inside* a
single byte.

And the pickers hold real data, not numbers: the **map's own scripts, by name** (`3 — Fisher 2`, out of
the cartridge), the real item list, the real trainer classes. Twilight asked for exactly this and the
data was already in `maps.json`; nothing needed extracting.

### The fifth sprite bug, and it was mine

`pictureIDCopy` is **not** a second copy of the picture id. I read the name, assumed it was, and shipped
a *"the game's second copy disagrees"* row — which fired on **every sprite in every save**, because the
game **wipes that byte**:

```asm
; the pictures IDs stored at [x#SPRITESTATEDATA2_PICTUREID] are no longer needed,
; so zero them
.zeroStoredPictureIDLoop
    xor a
    ld [hl], a
        -- engine/overworld/map_sprites.asm, end of LoadMapSpriteTilePatterns
```

It is scratch. At rest it is 0, always. **A warning that fires on every save is not a warning, it is
noise — and noise is a bug.** That is the second time in one day I have had to learn that (the ROM-cast
warning was the first). It is written into [`reference/sprites.md`](../../reference/sprites.md) as its
own section, in those words.

The screenshot review caught it. This is what the screenshot review is *for*.

### The rest of her list

* **Drag and drop, rewritten.** Qt's `Drag`/`DropArea` is **gone from this screen** and is not coming
  back — it broke twice, invisibly. A drop is now a **function call with a coordinate**:
  `canvas.dropCharacter(x, y, picture)`, `canvas.overDeleteZone(x, y)`. Plain containment tests you can
  read, reason about and test.
* **The player's drag was glitchy** because it accumulated a delta from the press point — which drifts,
  because the sprite's own coordinate moves under you while the press point stays put. It now asks
  *where the cursor is* (`tileAtGlobal`) and puts him there. Nothing to drift.
* **The map no longer resizes when a panel opens.** The docks always float. (They used to seat
  themselves in the row when there was room, which re-laid-out the canvas under the cursor.)
* **Music** is a real grouped `ComboBox` — the same control and the same delegate as the 248-map
  picker — with the **volume slider directly under it**. And **hover no longer auditions**: reading the
  list used to change what you were hearing.
* **The walls of text are gone.** Every panel's paragraph is now a **?** in its title bar
  (`MapInfoIcon`), and every byte the console recomputes on load wears a yellow **!** (`MapWarnIcon`).
  One of each per panel. *Don't litter them.*
* Layers: **Clear** in the panel title **and one per category** (a real button that says Clear, not a
  `⊘` on the tool chassis); *"none here"* deleted; **Meaning → Components**.
* The **✕** in the Details panel is a **Delete** button that says Delete.
* The Player is **safe** in the Characters panel now — `SPRITE_RED` is loaded on every map, ahead of the
  sprite set, so saying "this map hasn't loaded the player's picture" was nonsense. He carries a **?**
  instead: *drop one out and you get a second Red.*

### Two harness gotchas, both worth an hour of anyone's life

1. **`{"cmd":"screen"}` on a screen you are already on pushes a SECOND copy onto the router stack.**
   `list` then returns three `mapLeftPanel`s and `findItem` drives the first — a dead one, behind the
   live screen. Every `set` "succeeds" against an invisible copy and every screenshot looks unchanged.
2. **Hold one socket open across a whole run and every `shot` comes back byte-identical.** One
   connection per command and the grabs are live.

Both are now in [`reference/dev-harness.md`](../../reference/dev-harness.md).

## s16g — Three QML bugs, and everything on the screen was broken — `0.29.0-alpha`

Twilight's second pass. The list was long and every item was fair:

> *"The characters dont seem to move. Cant click on a character only click and drag to select them…
> Clicking edit does not bring up the side panel in edit mode… Dragging characters over and dropping
> them doesnt work. Removing in any way doesnt work."*

Five symptoms. **Three bugs**, all QML, all now in
[`reference/qt-patterns.md`](../../reference/qt-patterns.md) under a heading that says what they cost.

### 1. A binding on a C++ **method** never re-evaluates

`Repeater { model: brg.map.npcList() }` binds **once**, at load, and never again -- a QML binding
re-evaluates when a *property* it read changes, and a C++ method reads none that QML can see. So the
canvas went on drawing the cast it had memorised at startup.

That one bug is *four* of her symptoms: the simulation "didn't work" (it worked perfectly -- nothing
repainted), a dragged sprite snapped back, a delete vanished into nothing, a drop appeared to do nothing.
The C++ was right the whole time -- `tst_map_sprites` proves the sim moves the walkers and leaves the
`STAY` sprites alone, and it passed on the first run.

**This is the third time this shape has bitten in two days** (the blank Details panel was the same thing).
It is now written down as a rule: *if you bind to a C++ method, you need a revision counter.*

### 2. `Drag` works on GEOMETRIC OVERLAP -- so the source has to actually move

Qt's `DropArea` intersects itself with the **dragged item's own geometry**. I had a decorative ghost
following the cursor while the real `Drag.source` stayed parked inside the panel -- so the source was
never over the map and **nothing ever dropped**. Same bug in reverse for deleting: a sprite is clamped to
the map, so it could never reach the panel I was asking the user to drag it to.

The ghost **is** the source now. And the drop accepts with `Qt.MoveAction` specifically, so the sprite can
tell "I was deleted" from "I was let go over open ground" -- otherwise a delete also commits a move on the
thing it just destroyed.

### 3. An instance's `onFoo:` overrides the component's own `onFoo:`

`SimWarningDialog` remembered "don't show me this again" in its `onAccepted` -- and the use site declared
`onAccepted: brg.mapSim.playing = true`, which **replaced** it. The preference would never have stuck.

### The input rewrite

Sprites were TapHandler + DragHandler, and that combination produced three of her complaints at once: you
could not **click** a sprite (the drag handler took the grab first), the **✎** opened the panel in
map-details mode (the canvas's own TapHandler fired too and cleared the selection), and the mouse could
get **stuck** to a sprite. One `MouseArea` -- exclusive grab, consumes the event, unambiguous
press/release, `onCanceled` to clean up a stolen grab -- and all three go away.

**And the player is a MapSprite now too**: click him, drag him. There was never a reason he should be the
one thing on the map you could not pick up.

### The rest of her list

- **NO ATLAS.** She hates them and I never asked. 72 loose PNGs (`03_prof_oak.png`), which you can open
  and look at and replace. The load cost of 72 tiny files in a Qt resource is nil.
- **Music is a toolbar chip**, in the map picker's language: track list, sub-tracks toggle, the two save
  flags, a ▶. The music *panel* is gone -- a dock panel for one combo and two checkboxes was a panel too
  many.
- **Tooltips are gated on the "?" toggle.** The map screen had been ignoring it *on purpose* and that was
  wrong. One exception, explicitly granted: the **!** on a character this map cannot draw.
- The destructive-walk dialog **looks like a warning** now, not a blog page. Dark text, a stripe, a ⚠.
- The **!** badges are centred (`anchors.centerIn` centres a Label's *box*, and a "!" has no descender, so
  it sat bottom-left).

### And on the zoom

> *"whats wrong with sub-pixel zooming true smooth zoom not animation hacks"*

Right, and the answer is: **nothing, for any input that is actually continuous.** The slider, a trackpad's
`pixelDelta`, a pinch -- those are sub-pixel already and now go **straight through with no interpolation
at all**. The one thing that cannot be continuous is a **mouse wheel detent**: the hardware reports 120
units per click and there is *nothing in between*. That single case gets a 90ms bridge. Nothing else does.

Full `ctest` **83/83**, with `tst_map_sprites` up to 11 cases (the new one drives the simulation by hand
and demands the walkers walk and the `STAY` sprites do not).

## s16f — Twilight's review, and the screenshots that had no text in them — `0.28.0-alpha`

> *"first of all in your screenshots theres no text so im assuming you didnt scrutinize very hard on
> the images lol"*

She was right, and it is the most important thing in this entry. Every screenshot I reviewed the map
screen with was **tofu boxes where the words should be** — because the app's `--shot` flag does not set
`QT_QPA_FONTDIR` and I never set it. So I reviewed the *layout* and told myself I had reviewed the
*screen*. The notes already documented the trap (`reference/screenshots.md`) and I did not read them
carefully enough.

Fixed, and the warning is now the **first thing in that file**: if the picture has boxes where the
labels should be, stop and fix the capture before looking at anything else.

With text on, the very first honest look found three things at once: the old −/+/fit zoom buttons still
sitting in the status bar, two **identical ▶ buttons** side by side in the toolbar, and a zoom menu that
ran off the bottom of the default window.

### The smooth zoom, done properly

The zoom snapped to whole numbers because a fractional scale ruins pixel art. The fix is a **shader** —
`shaders/pixelart.frag`, anti-aliased point sampling: snap the sample onto the texel centre but leave a
ramp exactly **one screen pixel** wide across the seam (`fwidth` is what makes it exactly one, at any
zoom). Flat inside a pixel, a single pixel of blend at the edge. Crisp like nearest, smooth like
bilinear, and **pixel-identical to nearest at whole zooms**, so nothing was given up.

Verified on the real GPU at a deliberately awkward **2.37×**: clean, uniform, sharp. No ripple.

⚠️ **And it caught its own trap.** `ShaderEffect` **does not run on Qt Quick's software backend** — it
silently draws nothing, and the first headless screenshot of the map came back **completely black**.
Every headless run in this project is software-backed (the screenshooter, the GUI suites,
`tst_visual_regression`). `PixelImage.qml` now falls back to plain nearest when `GraphicsInfo.api` is
`Software`. That is not a fudge: nearest is the honest behaviour of the only sampler that backend has,
and it is exactly right at the whole zooms a test uses.

### The rest of her list

- **Opens where the player is** — the Game Boy's screen plus one block, him centred. Not a postage
  stamp of the whole route. And **no panels open**: you open the Map screen and you see the map.
- **Zoom has ONE home** — a **▾** on the zoom tool: a log-scale slider, and a **Go to…** list (the map,
  the camera, the player, a random person / object / door / warp / block / connecting route). A
  destination this map hasn't got is greyed and says *"none here"*. The status bar keeps the number and
  loses the buttons — a number is a fact, and facts are what that bar is for.
- **The wheel and the slider move in RATIOS**, not fixed steps. A "+1" bite is violent at 0.6× and
  glacial at 10×; a constant ratio feels the same everywhere, which is the whole point of a smooth zoom.
- **The Characters panel is IN the left dock.** I had made it a second rail stacked beside the dock —
  which is precisely the thing this screen was rebuilt to abolish, and it is written in the notes I was
  supposed to be working from. My mistake, and a plain one.
- It now leads with **"Safe on this map"** (ungrouped, the pictures this map has actually loaded), with
  the full grouped list underneath. Dots became **!**; the tooltip fires **on the !** and is
  white-on-black instead of dark text on a see-through box; the filter box lost its huge margin.
- **▶ Walk** — a labelled chip, not a second bare ▶, because two identical play buttons side by side is
  a puzzle rather than a toolbar. The NPCs wander according to their own movement bytes. **Destructive
  on purpose** (see `MapSim`'s note on why a shadow world would be worse), announced once, with a
  "don't show again" that starts **unticked**.
- **Everything paused by default** — the animation, the walking, the music.

Full `ctest` **83/83**.

## s16e — Phase 4 finished: 4b's editing half, the Characters bar, the Details panel — `0.27.0-alpha`

The map screen stopped being a picture. You can build a town's cast with the mouse now: drag a
character in from the rail, drag them around, drag them back out to delete, and edit every byte they
have.

Three UI mistakes I made and had to fix, all caught by *actually looking at the screenshots*:

**1. The Details panel came up permanently blank — and every C++ test passed.** The panel is created by
the dock's `Loader`, which hands it the canvas in `onLoaded` — *after* `Component.onCompleted`. So my
imperative `refresh()` ran once, with `canvas` still null, decided nothing was selected, and was never
called again. The C++ was returning 28 fields the whole time; the QML simply never asked. Fixed by
making `sprite` and `fields` **bindings** (with a `revision` counter so an edit re-asks) instead of an
imperative refresh. The lesson is the old one: in QML, if you find yourself calling a refresh function,
ask why a binding wouldn't do.

**2. The number boxes fell off the right edge of the panel.** `label | combo | number` on one row wants
~250 px; a dock panel is ~190. Shrinking the controls to fit would have been shrinking the *content* to
make room for the *furniture* — exactly backwards. The label moved above its controls.

**3. The Characters bar drew 16×16 art at 24 px.** A 1.5× scale on pixel art is mush. It's a whole 2×
now. And the bar collapses to a strip, because at the default window it, the Layers dock and the map
cannot all have what they want — and the map is not the thing that gives way.

### ⚠️ And a *design* bug the test caught, which is the interesting one

The Details panel owes the user a sentence: *the game rebuilds this map's original cast from the
cartridge the moment the player walks back in* (the console proved that this morning — s16a). My first
version decided **when** to say it by comparing the save's cast against the ROM's.

`tst_map_sprites` failed on the very first run, on an **untouched** save. And it was right:

> **maps.json says Pallet Town's Girl stands at (3, 8). `BaseSAV` has her at (3, 6).**

She's a `WALK` sprite. **She had wandered.** A real save's cast *always* differs from the cartridge's,
because the walking NPCs move around before you save. The warning would have fired on essentially every
save anyone ever opened — and a warning that always fires is noise, and noise is a bug.

So the panel now tracks the **edit**, not the difference: quiet until *you* change something.

Ten cases in the new **`tst_map_sprites`**, and its keystone is
`moveNpc_writesExactlyTwoBytes`: drag a person across town, then byte-diff the **whole 32 KB save** and
demand that **exactly `mapX` and `mapY`** moved. Nothing else. That is the byte-fidelity rule with
teeth.

Full `ctest` **83/83**.

## s16c — The map has people on it (4b, the rendering half) — `0.26.0-alpha`

Every town in the editor has been empty since the map screen was built, and the reason turned out to be
embarrassingly simple: the repo carried **one** sprite sheet. `red.png`. The player. The other
seventy-one were never imported, so there was nothing to draw.

`scripts/import_sprites.py` now pulls **all 72** out of `pret/pokered` — one atlas
(`assets/sprites/overworld.png`, 72 cells of 16×96) plus a generated `spriteart.h` frame table. It is
self-validating and it **refused my first version**, which is exactly what it is for: I had assumed two
art shapes (16×96 walkers, 16×16 objects) and it came back with twenty files at **16×48**.

That third shape is real and it is the other half of the Game Boy's sprite budget: **twenty "still"
people** — the nurses, the guards, Mom, the Gameboy Kid — who have **three frames and no walking art at
all**, because the game never draws them taking a step.

What landed:

- **`MapEngine::npcSprite()`** — any picture id, any facing, through the OBJECT palette. Right is
  **left, mirrored** (there is no right-facing art for *anybody*), colour 0 transparent, a one-frame
  object resolves every facing to its single quad.
- **`MapModel::npcList()`** — every used slot, with the `+4` bias taken off, and **`inSpriteSet`**: is
  this map actually carrying that sprite's picture? When it isn't, the canvas outlines it in **amber** —
  that is the sprite the console draws as garbage, and she should hear it from us, not from the
  cartridge.
- **A *People & objects* layer**, on by default.
- **The ground is no longer selectable.** Clicking a block did nothing useful and only fought the thing
  that *should* be clickable. **Sprites are now the only selectable object on the map.**

Verified by eye on Pallet Town: **Oak at (8,5), the Girl at (3,6), the Fisher at (11,14)** — every one
where the console's own RAM says they are. Full `ctest` 82/82.

**Still owed on 4b:** drag-to-move, and the **✕ delete** / **✎ edit** buttons on a selected sprite.
Then 4c (the Characters bar) and 4d (the Details panel).

## s16b — The four bugs, fixed — `0.25.1-alpha`

Every fix is pinned by a test, and every test was **negative-controlled**: the bug was put back, one at
a time, and each case failed *by name*, with the byte.

- `SpriteMobility` un-inverted (`STAY = $FF`, `WALK = $FE`).
- `load(MapDBEntrySprite*)` no longer writes **WALK for a STAY sprite**.
- `SpriteGrass` un-inverted (`$80` = in grass) — `reset()` had been flagging every blank sprite as
  standing in a field.
- **`face` and `range` are one byte**, and it now goes to one place. A test walks all ~918 map sprites
  and a second one proves movement byte 2 can never again land in the *facing* field.
- The five fields the game keeps and we never read — Y/X-adjusted, collision data, original facing, the
  duplicate picture id — are modelled and round-trip. The bytes the game *doesn't* use are still left
  untouched, and a test says so.
- `sprites.json` gained a curated **`group`** (Story / Trainers / Townsfolk / Pokémon / Objects), pinned
  by `tst_db_integrity`.

**And a correction I owe the record.** The research draft claimed we didn't model the missable
visibility flags at all, and put them at `0x28A0`. **Both were wrong.** `WorldMissables` has read them
from `0x2852` since 2020 — the flags sit *before* the list in WRAM. I caught it only because
`getBitField` turned up in a grep. The real gap is smaller and different: **nothing wires a sprite to
its flag**, and the polarity (*bit set = hidden*) was written down nowhere. Both now fixed in the docs.

## s16a — Sprites researched: four bugs, four gaps, and the console overruling me

Twilight, opening the map screen's next phase: *"Sprites are complex and detailed and a lot of information
on pokered-save-editor could be wrong or missing a lot of information."* She was right on both counts, and
more so than she knew.

The full write-up is in [`reference/sprites.md`](../../reference/sprites.md) (Parts 3, 5 and 6 are new).
The design it produced is [`plans/map-screen.md`](../../plans/map-screen.md) → **Phase 4**, rewritten from
scratch into four sub-phases.

### v1 is not a source. It is the same code.

`assets/references/pokered-save-editor/.../SpriteData.ts` and our `spritedata.cpp` are the **same model** —
same offsets, same split, same optionals, same `saveMissables`. There is nothing to *learn* from v1 here,
because v2 already is v1. Its UI is a plain accordion of sprite cards; there is nothing to carry over.

The one thing v1 had going for it: **no enums.** Its inline comments — *"0xFF not moving, 0xFE random
movements"*, *"0x80 in grass"* — are **correct**. The enums are a **v2 invention**, and they are backwards.

### Four bugs, three of them proven on the cartridge

| | | |
|---|---|---|
| **`SpriteMobility` is inverted** | says `Moving = 0xFF, NotMoving = 0xFE` | ROM: **`STAY = $FF`, `WALK = $FE`**. Console: Pallet's Oak (`STAY`) reads `$FF`; the Girl and the Fisher (`WALK`) read `$FE`. |
| **`setTo()` writes the wrong one** | `"Stay"` → `NotMoving` (`0xFE`) | it writes **WALK for a STAY sprite**. Every `setTo()` and every randomize has been getting it backwards. |
| **`SpriteGrass` is inverted** | `InGrass = 0x00, NotInGrass = 0x80` | `wram.asm`: **`$80` = in grass**. So `reset()` has been flagging **every blank sprite as standing in grass**. |
| **`face` and `range` are ONE byte** | `face` → `faceDir`, `range` → `rangeDirByte` | both are **movement byte 2** (`wMapSpriteData`). `faceDir` is a *different field in a different table*. Console: Oak's movement byte 2 is `$FF` (= `maps.json` `"face":"None"`), the Girl's is `$00` (= `"range":0`). |

Plus four things never modelled at all: StateData1 `a`/`b`/`c` (Y-adjusted, X-adjusted, collision data),
StateData2 `9` (original facing) and `d` (duplicate picture id), and — the big one —
**`wToggleableObjectFlags`** (`0x28A0`, 32 bytes, *bit set = hidden*). That flag array is what actually
decides whether a missable NPC is on the map. We model only the per-map **list** (`0x287A`), which the game
**rebuilds from ROM on every map load** and which therefore does nothing at all.

### And then the cartridge told me I was wrong

Reading `LoadMapHeader` (`.loadSpriteData`) is alarming: on a map load it **zeroes `$F0` bytes of both
sprite tables** for slots 1–15 and reloads picture / X / Y / movement from the **ROM**, unless
`wStatusFlags4` bit 5 is set — which it isn't in a normal save. I read that and concluded, with some
confidence, that **every sprite byte an editor writes is erased the moment the game boots** — that the whole
feature was writing into a bin, exactly like the sprite-set cache.

So I wrote `scripts/emu/probe_sprite_persistence.py`, booted the real cartridge with three saves, and read
back the console's own WRAM.

**The edits survived.** All of them. Slot 1 re-pictured and moved, a **fourth NPC invented out of nothing**
in a town that has three — `wNumSprites 4`, `pic $2E` at `(9,9)`, `pic $05` at `(6,11)`, straight off the
console. The status bit changed nothing.

The reason is one line in `MainMenu`: `.choseContinue` sets **`BIT_CUR_MAP_LOADED_1`** before jumping to
`SpecialEnterMap`, so the map header is **never re-read** on the continue path. The saved sprite state is
what the player walks into.

The honest statement — the one the screen will make out loud — is therefore:

> An edited sprite **is really there** when the save loads, and the game **restores the map's original cast
> from ROM the moment the player leaves and walks back in.**

Which is precisely what Twilight observed in v1 years ago (*"sprite edits work; outdoor sprites are
glitchy"*), and precisely the derived-byte doctrine: **show it, never hide it, never silently fix it.**

I would have shipped the wrong conclusion off a careful reading of the source. The rule holds: **the console
is the judge.**

### The design that came out of it

Twilight's calls, taken this session:

- **Background squares stop being selectable.** Blocks and tiles belong in their panels; the canvas should
  not compete with them. **Sprites are, for now, the only selectable thing on the map.**
- The "Sprite panel" is a **Details panel**, on the **left** — it edits whatever is selected, and with
  **nothing selected it shows the map's own details**. Never blank.
- A **Characters bar** (separate from the sprite-*set* panel): 1–2 columns, grouped/sorted/filterable,
  **drag a character onto the map to add it, drag a sprite back out to delete it**.
- A selected sprite carries a **✕ delete** and a **✎ edit** button; ✎ opens the Details panel on it.
- Deleting **slides the rest up** ("isn't that obvious").
- `sprites.json` gains a curated **`group`** field (People / Trainers / Pokémon / Objects) — **to be
  approved before it is written**.
- Sized as **four sub-phases**: **4a** the data model made true · **4b** NPCs drawn + selectable + draggable
  · **4c** the Characters bar · **4d** the Details panel.

Nothing is built yet. 4a comes first, deliberately: **nothing else may be built on a lie.**
