## s16a — Sprites researched: four bugs, four gaps, and the console overruling me

Twilight, opening the map screen's next phase: *"Sprites are complex and detailed and a lot of information
on pokered-save-editor could be wrong or missing a lot of information."* She was right on both counts, and
more so than she knew.

The full write-up is in [`reference/sprites.md`](../../reference/sprites.md) (Parts 3, 5 and 6 are new).
The design it produced is [`plans/map-screen.md`](../../plans/map-screen.md) → **Phase 4**, rewritten from
scratch into four sub-phases.

### v1 is not a source. It is the same code.

`assets/references/pokered-save-editor/.../SpriteData.ts` and our `spritedata.cpp` are the **same model** —
same offsets, same split, same optionals, same `saveMissables`. There is nothing to *learn* from v1 here,
because v2 already is v1. Its UI is a plain accordion of sprite cards; there is nothing to carry over.

The one thing v1 had going for it: **no enums.** Its inline comments — *"0xFF not moving, 0xFE random
movements"*, *"0x80 in grass"* — are **correct**. The enums are a **v2 invention**, and they are backwards.

### Four bugs, three of them proven on the cartridge

| | | |
|---|---|---|
| **`SpriteMobility` is inverted** | says `Moving = 0xFF, NotMoving = 0xFE` | ROM: **`STAY = $FF`, `WALK = $FE`**. Console: Pallet's Oak (`STAY`) reads `$FF`; the Girl and the Fisher (`WALK`) read `$FE`. |
| **`setTo()` writes the wrong one** | `"Stay"` → `NotMoving` (`0xFE`) | it writes **WALK for a STAY sprite**. Every `setTo()` and every randomize has been getting it backwards. |
| **`SpriteGrass` is inverted** | `InGrass = 0x00, NotInGrass = 0x80` | `wram.asm`: **`$80` = in grass**. So `reset()` has been flagging **every blank sprite as standing in grass**. |
| **`face` and `range` are ONE byte** | `face` → `faceDir`, `range` → `rangeDirByte` | both are **movement byte 2** (`wMapSpriteData`). `faceDir` is a *different field in a different table*. Console: Oak's movement byte 2 is `$FF` (= `maps.json` `"face":"None"`), the Girl's is `$00` (= `"range":0`). |

Plus four things never modelled at all: StateData1 `a`/`b`/`c` (Y-adjusted, X-adjusted, collision data),
StateData2 `9` (original facing) and `d` (duplicate picture id), and — the big one —
**`wToggleableObjectFlags`** (`0x28A0`, 32 bytes, *bit set = hidden*). That flag array is what actually
decides whether a missable NPC is on the map. We model only the per-map **list** (`0x287A`), which the game
**rebuilds from ROM on every map load** and which therefore does nothing at all.

### And then the cartridge told me I was wrong

Reading `LoadMapHeader` (`.loadSpriteData`) is alarming: on a map load it **zeroes `$F0` bytes of both
sprite tables** for slots 1–15 and reloads picture / X / Y / movement from the **ROM**, unless
`wStatusFlags4` bit 5 is set — which it isn't in a normal save. I read that and concluded, with some
confidence, that **every sprite byte an editor writes is erased the moment the game boots** — that the whole
feature was writing into a bin, exactly like the sprite-set cache.

So I wrote `scripts/emu/probe_sprite_persistence.py`, booted the real cartridge with three saves, and read
back the console's own WRAM.

**The edits survived.** All of them. Slot 1 re-pictured and moved, a **fourth NPC invented out of nothing**
in a town that has three — `wNumSprites 4`, `pic $2E` at `(9,9)`, `pic $05` at `(6,11)`, straight off the
console. The status bit changed nothing.

The reason is one line in `MainMenu`: `.choseContinue` sets **`BIT_CUR_MAP_LOADED_1`** before jumping to
`SpecialEnterMap`, so the map header is **never re-read** on the continue path. The saved sprite state is
what the player walks into.

The honest statement — the one the screen will make out loud — is therefore:

> An edited sprite **is really there** when the save loads, and the game **restores the map's original cast
> from ROM the moment the player leaves and walks back in.**

Which is precisely what Twilight observed in v1 years ago (*"sprite edits work; outdoor sprites are
glitchy"*), and precisely the derived-byte doctrine: **show it, never hide it, never silently fix it.**

I would have shipped the wrong conclusion off a careful reading of the source. The rule holds: **the console
is the judge.**

### The design that came out of it

Twilight's calls, taken this session:

- **Background squares stop being selectable.** Blocks and tiles belong in their panels; the canvas should
  not compete with them. **Sprites are, for now, the only selectable thing on the map.**
- The "Sprite panel" is a **Details panel**, on the **left** — it edits whatever is selected, and with
  **nothing selected it shows the map's own details**. Never blank.
- A **Characters bar** (separate from the sprite-*set* panel): 1–2 columns, grouped/sorted/filterable,
  **drag a character onto the map to add it, drag a sprite back out to delete it**.
- A selected sprite carries a **✕ delete** and a **✎ edit** button; ✎ opens the Details panel on it.
- Deleting **slides the rest up** ("isn't that obvious").
- `sprites.json` gains a curated **`group`** field (People / Trainers / Pokémon / Objects) — **to be
  approved before it is written**.
- Sized as **four sub-phases**: **4a** the data model made true · **4b** NPCs drawn + selectable + draggable
  · **4c** the Characters bar · **4d** the Details panel.

Nothing is built yet. 4a comes first, deliberately: **nothing else may be built on a lie.**
