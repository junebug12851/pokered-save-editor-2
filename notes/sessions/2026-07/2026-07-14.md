# 2026-07-14

## WARPS — the research pass. The console overruled us on one thing and confirmed us on ten.

Twilight asked for warps on the map screen, and asked the right question first: *"I'm pretty lost on
what most of the other ones even are. Actual useful names and descriptions would be great… and it would
be wonderful to know which ones were regenerated or rewritten on save load."*

So nothing was built. The research pass ran instead, out of `pret/pokered` and then **against the
cartridge**. Everything is in **[`../../reference/warps.md`](../../reference/warps.md)**; the design it
feeds is **[`../../plans/map-screen.md`](../../plans/map-screen.md) → Phase 5** (5a–5e).

New tool: **`scripts/emu/probe_warp_persistence.py`** — boots the real ROM with three tampered saves and
reports, byte by byte, what the console keeps and what it rewrites.

### The linchpin — why an edited warp works at all

`LoadMapHeader` rebuilds the warp list **from ROM** on every map load, and — unlike the sprite block —
it has **no escape-hatch bit**. On the face of it every warp we write is erased on load.

It isn't, and the reason is one line in `LoadMainData` (`engine/menus/save.asm`), the routine that reads
the save off the cartridge:

```asm
	ld hl, wCurMapTileset
	set BIT_NO_PREVIOUS_MAP, [hl]   ; bit 7
```

…which makes `LoadMapHeader` **`ret nz` before it copies anything.** The game flags *"there is no
previous map"* as it loads your save, and the next header load bails out. **Our warps are the ones the
console runs on** — and the game restores the map's original doors the moment the player leaves and
walks back in. The sprite story exactly, by a different mechanism.

The console agrees. Warp 0 moved to (2,2), re-aimed at Viridian Forest, and **a 4th warp invented in a
3-warp town**:

```
  B  warp LIST tampered
    wNumberOfWarps  4
      warp 0   tile=( 2, 2)   -> map 0x2A, warp 3     <- ours
      warp 3   tile=( 9, 9)   -> map 0x25, warp 0     <- invented out of nothing
  VERDICT
    Does the console run on OUR warp list?   YES -- the save wins on Continue
```

### 🟡 The regenerated ones — and it's a whole byte, not two bits

Twilight wanted the rewritten-on-load fields marked. The answer is sharper than expected.

`wStatusFlags3` has an **alias**: `wCableClubDestinationMap` is *the same byte*. And `SpecialEnterMap` —
which is on the Continue path — does `xor a / ld [wCableClubDestinationMap], a`. **The entire byte is
zeroed on every save load.** Written `$FF`, booted the real ROM, read back **`0x00`**.

So these can *never* survive a save, and every one of them gets the **!**:

| bit | Where we keep it | Real name |
|---|---|---|
| 0 | `AreaNPC::tradeCenterSpritesFaced` | `BIT_INIT_TRADE_CENTER_FACING` |
| 3 | `AreaWarps::scriptedWarp` | `BIT_WARP_FROM_CUR_SCRIPT` |
| 4 | `AreaWarps::isDungeonWarp` | `BIT_ON_DUNGEON_WARP` |
| 5 | `AreaNPC::npcsFaceAway` | `BIT_NO_NPC_FACE_PLAYER` |
| 6 | `AreaPlayer::isBattle` | **`BIT_TALKED_TO_TRAINER`** |
| 7 | `AreaPlayer::isTrainerBattle` | **`BIT_PRINT_END_BATTLE_TEXT`** |

Every *other* warp byte we tampered with came back **exactly as written**. So the `!` list is short and
certain — which is the whole point of asking the console rather than guessing.

### 💀 Two of v1's fields are dead

`wWarpedFromWhichWarp` and `wWarpedFromWhichMap` — v1's *"From Warp"* and *"From Map"*. `WarpFound2`
writes them on every warp. **Nothing in the entire game reads either one.** Two writes, zero reads,
across the whole disassembly. They survive a save perfectly; they are simply **inert** — a breadcrumb
the game drops and never picks up.

⚠️ **A wiped byte and an unread byte are not the same fact**, and the panel must say which is which.

### ⭐ …and the two that actually matter weren't on the warps screen at all

Twilight's instinct — *"from map needs to be in the top toolbar, it's pretty important to workflow"* —
is right; the byte named that is the dead one. The field she is reaching for is **`wLastMap`**
(`0x2611`): **the map that a `$FF` door returns you to.** Every building's exit is `$FF`. Walk out of
Red's house and you are in Pallet Town *because this byte says Pallet Town*. Its partner is
**`wLastBlackoutMap`** (`0x29C5`) — where blacking out, **Dig** and **Escape Rope** put you.

**Both are already modelled** (`WorldGeneral::lastMap` / `lastBlackoutMap`) — no new C++. They just have
to be *visible to the person editing warps*. So `wLastMap` goes in the **top toolbar**, in words —
**`[ Outside is: Pallet Town ⌄ ]`** — because changing it re-labels every `$FF` door on the map at once.

### 🔫 Two loaded guns — and we are currently loading them ourselves

Same class as the music bank: a save value the console will use to read arbitrary ROM.

- **`wDestinationMap`** (Fly / special warp) is looked up in `FlyWarpDataPtr` — **13 entries, no
  terminator, no bounds check.** Any other map id runs the loop off the end of the table and copies six
  arbitrary ROM bytes into the view pointer and the player's coordinates.
- **`wDungeonWarpDestinationMap` + `wWhichDungeonWarp`** must be one of **12 legal (map, hole) pairs**,
  and the hole number is **1-based**. Victory Road 2F has no hole 1.

🐞 **And `AreaWarps::setTo()` / `randomize()` fill both with random illegal values today** — any cave, any
map, a 0-based hole index. Dormant only because `MapsDB` is never deep-linked at boot; **it goes live the
moment that landmine is defused.** Fixed in Phase 5a, exactly as `AreaAudio::setTo()` was fixed before
*its* path went live.

### The other model bugs found (seven, all in Phase 5a)

`blackoutDest` is not a destination — it is **`BIT_ESCAPE_WARP`** (Dig / Escape Rope / blackout), and it
is sitting in `AreaMap` as a bool doc'd *"may be unused"*. `skipJoypadCheckWarps` is **`BIT_FORCED_WARP`**
(doors fire without walking into them — it's how the Seafoam current sweeps you along). And `destWarp`
indexes the destination map's **`warps_to` arrival list**, not its warp list — the thing everyone gets
wrong.

### The design (Phase 5, five sub-phases)

**5a** the model made true · **5b** warps + signs on the canvas (select/drag/add/delete, on the sprite
machinery) · **5c** **the toolbar becomes tools** — Twilight's ask: `⇄+ Place warp`, `🧍+ Place sprite`
(random picture *from the map's own loaded set*), `🪧+ Place sign`, each a real tool with a cursor, a
context bar and a keyboard path — plus `[ Outside is: … ]` · **5d** the **Warp State** group in the
Details panel, every byte named in English, the two guns offering only legal values (full range behind a
switch, never refused, never rewritten), and the four dead/wiped fields **collapsed behind a switch with

### Standing rule added to `CLAUDE.md`

**"RESEARCH LANDS IN THE NOTES — every time, by default."** Twilight asked for it explicitly. The shape
of a research pass is now written down: primary source → **ask the console when it matters** → the
reference note → *what it means for our code* (every pass so far has found real bugs: sprites four,
tilesets three, warps seven) → wire it into the nav/status/session/plan. **Then** design the UI. Three
new rows in the notes-maintenance trigger table.

### ⛔ …and then the scope got pulled back in, correctly

> Twilight: *"Let's not get too far ahead of ourselves. Signs and stuff, connecting routes, wild Pokémon —
> these are examples of things I haven't gotten to yet. I'd hate to have to undo a lot of work because it
> was done before I explained anything."*

She caught a real overreach. **Signs had walked into Phase 5b** — not because she asked for them, but
because they load out of the *same ROM block* as warps and are the *same shape*. That is adjacency, not a
brief, and it is exactly how a screen ends up getting built twice. A **🪧 Place sign** tool had also
appeared in the toolbar next to the two she actually named.

**Cut.** Signs are out of Phase 5. The maker rail is **exactly the two tools she named** (`⇄+ Place warp`,
`🧍+ Place sprite`) and is designed to grow one slot at a time as each object type gets its own
conversation.

New section: **`plans/map-screen.md` → §12b, "NOT YET BRIEFED"** — signs, connections ("connecting
routes"), wild Pokémon/encounters, area state, the tileset deep pass. The old sketches for those phases
were written **from the save layout**, which is a map of what bytes exist, not of what she wants a person
to be able to *do*, and they now say so explicitly: **they carry no authority.**

And a new standing rule in `CLAUDE.md` → *Critical Things Not to Get Wrong*:

> **DON'T BUILD WHAT TWILIGHT HASN'T BRIEFED — adjacency is not a brief.** A feature gets its own
> conversation first, then research, then a design, then code. A phase does **not** get to absorb a
> neighbouring feature because the data happens to sit next to it in the save. When a briefed feature
> genuinely needs an un-briefed one, it **reads** it; it does not build a UI for it. **Ask before
> building, not after.**
