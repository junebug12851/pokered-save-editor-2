# 2026-07-14

## THE PLAYER — briefed, researched, console-verified. Ten of 26 bytes die on load.

Twilight briefed **fleshing out the full player details panel**: all 26 `AreaPlayer` map-state bytes
(the v1 `area-player` field set — Moving, Last Stop, Current direction, coords + block coords, Jumping
Y, Strength/Fly/Surf, the battle four, the special-warp offsets, the standing-on trio, Walk/Bike/Surf,
End-edge jump, Spin, Card Key, Link Cable). Her ask: *"figure out what all those options are, figure
out which are rewritten on startup, organize structure, follow the ui/ux."*

Research first. The v2 model's **offsets and bit numbers are all correct** (it's a straight v1 port),
but the *names lie* and nothing said which the game keeps. So I read the load path in `pret/pokered`
and then **asked the cartridge** — new probe `scripts/emu/probe_player_state.py`, booting the real ROM
twice (battle-over bit clear vs set) and reading back all 26.

The answer to *"which are rewritten on startup"*: **ten are.**

- ⚠️ **Current direction (`wPlayerDirection`) is forced to DOWN on every Continue** — `MainMenu`
  `.pressedA` writes `PLAYER_DIR_DOWN` before `SpecialEnterMap`. (Probe: wrote UP `0x08`, read `0x04`.)
- ⚠️ **`wStatusFlags3` zeroed whole** → `isBattle`/`isTrainerBattle` (really `BIT_TALKED_TO_TRAINER` /
  `BIT_PRINT_END_BATTLE_TEXT`) die — the same `SpecialEnterMap` wipe warps/NPCs already ride.
- ⚠️ **Strength reset on an ordinary Continue, but SURVIVES if battle-over is set** — `EnterMap`
  branches on `BIT_BATTLE_OVER_OR_BLACKOUT`; the two-save probe caught the interaction.
- ⚠️ battle-over, link-cable, standing-on-door, moving-through-door, ledge/fishing, and Jumping Y all
  cleared/zeroed on entry.
- ❌ **Three dead:** `x/yOffsetSinceLastSpecialWarp` (`; they don't seem to be used`) and `usedCardKey`
  = `BIT_UNUSED_CARD_KEY` (`; never checked`).
- Five **misnamed** (fixed in 5f-0, before UI): the two battle bits, `flyOutofBattle`=`BIT_USED_FLY`,
  `finalLedgeJumping`=`BIT_LEDGE_OR_FISHING`, `usedCardKey`=`BIT_UNUSED_CARD_KEY`.

⚠️ **The probe was worth running:** `wMovementFlags` clears bits **per-routine, not whole-byte** (0/1/6
cleared on load, 2/7 kept) — a read of the asm alone would have called it a unit and been wrong, the
exact sprite-pass mistake.

Written up in **[`../../reference/player-state.md`](../../reference/player-state.md)**; design in
**[`../../plans/map-screen.md`](../../plans/map-screen.md) → Phase 5f** (5f-0 model rename · 5f-1 four
buckets · 5f-2 the `[ ⚠ Show N fields the game rewrites on load ]` switch · 5f-3 byte-diff test). No app
code yet — research + design, following the warp/sign precedent.

## SIGNS — briefed, researched, designed. And for once the model was already right.

Twilight briefed signs (held until now in `map-screen.md` §12b): *"add sign features — an add-sign
tool, and the details panel needs x,y and an actual combo box to select from the text on the map; the
usual delete options, much like warps and sprites."* On the combo she wanted **all** text ids a sign
can reference, **grouped**; on scope, *"break it into as many phases as you need."*

Research first, out of `pret/pokered`. Findings in **[`../../reference/signs.md`](../../reference/signs.md)**;
design in **[`../../plans/map-screen.md`](../../plans/map-screen.md) → Phase 6** (6a–6d), which bumped
the old placeholder phases 6–13 down to 7–14.

- **The save model is already correct** — the rare pass with *no bug to fix first*. `AreaSign`
  (`wNumSigns` `0x275C`, cap 16 = `MAX_BG_EVENTS`) and `SignData` (`wSignCoords` Y,X `0x275D`;
  `wSignTextIDs` `0x277D`) match the cartridge. So there is no "6a: make the model true"; 6a is *data*.
- **The words aren't in our data.** A sign's `text` is a **1-based index into the map's
  `def_text_pointers` table** (verified: Pallet Town id 4 → `_PalletTownOaksLabSignText` = "OAK #MON /
  RESEARCH LAB", which is exactly what `maps.json` carries). `maps.json` ships only the id, so **6a**
  extracts the real strings from `pret/pokered` and groups every text id as sign / person / script-only.
- **Same persistence linchpin as warps** — `.loadSignData` sits inside `LoadMapHeader`, behind
  `BIT_NO_PREVIOUS_MAP`; one guard protects warps, signs and sprites together. Edited signs are live on
  Continue, restored on re-entry — the panel reuses the warp note.

**Built, same session — all four phases (`0.38.0-alpha`).** 6a: `scripts/import_sign_text.py` pulls the
real words out of `pret/pokered` into `maps.json` (additive-only — strip `textEntries` and the file is
byte-identical), plus `MapDBEntryText` + `MapDBEntry::getTextEntries()`. 6b: orange `▤` chips on a new
Signs layer, select/drag(2 bytes)/delete/edit. 6c: the `▤` Place sign tool (shortcut S). 6d: the Details
panel — one X/Y control + the grouped **Says…** picker of the map's real text, Delete, the live-on-
Continue note. `tst_signs` 15/15 (keystone: drag byte-diffs the whole save to exactly the two coord
bytes; `everyShippedSignResolvesInItsMapsText` proves the data). `tst_warps` 24/24, `tst_map_layers`
12/12, `tst_qml_screens` 16/16, `tst_qml_brg` 11/11 all green. Screenshot-reviewed on the live build
(Pallet Town: sign chips draw, selection shows the words, the panel resolves "OAK POKéMON / RESEARCH
LAB"). Owed: Twilight's live pass.

## WARPS — the research pass. The console overruled us on one thing and confirmed us on ten.

Twilight asked for warps on the map screen, and asked the right question first: *"I'm pretty lost on
what most of the other ones even are. Actual useful names and descriptions would be great… and it would
be wonderful to know which ones were regenerated or rewritten on save load."*

So nothing was built. The research pass ran instead, out of `pret/pokered` and then **against the
cartridge**. Everything is in **[`../../reference/warps.md`](../../reference/warps.md)**; the design it
feeds is **[`../../plans/map-screen.md`](../../plans/map-screen.md) → Phase 5** (5a–5e).

New tool: **`scripts/emu/probe_warp_persistence.py`** — boots the real ROM with three tampered saves and
reports, byte by byte, what the console keeps and what it rewrites.

### The linchpin — why an edited warp works at all

`LoadMapHeader` rebuilds the warp list **from ROM** on every map load, and — unlike the sprite block —
it has **no escape-hatch bit**. On the face of it every warp we write is erased on load.

It isn't, and the reason is one line in `LoadMainData` (`engine/menus/save.asm`), the routine that reads
the save off the cartridge:

```asm
	ld hl, wCurMapTileset
	set BIT_NO_PREVIOUS_MAP, [hl]   ; bit 7
```

…which makes `LoadMapHeader` **`ret nz` before it copies anything.** The game flags *"there is no
previous map"* as it loads your save, and the next header load bails out. **Our warps are the ones the
console runs on** — and the game restores the map's original doors the moment the player leaves and
walks back in. The sprite story exactly, by a different mechanism.

The console agrees. Warp 0 moved to (2,2), re-aimed at Viridian Forest, and **a 4th warp invented in a
3-warp town**:

```
  B  warp LIST tampered
    wNumberOfWarps  4
      warp 0   tile=( 2, 2)   -> map 0x2A, warp 3     <- ours
      warp 3   tile=( 9, 9)   -> map 0x25, warp 0     <- invented out of nothing
  VERDICT
    Does the console run on OUR warp list?   YES -- the save wins on Continue
```

### The regenerated ones — and it's a whole byte, not two bits

Twilight wanted the rewritten-on-load fields marked. The answer is sharper than expected.

`wStatusFlags3` has an **alias**: `wCableClubDestinationMap` is *the same byte*. And `SpecialEnterMap` —
which is on the Continue path — does `xor a / ld [wCableClubDestinationMap], a`. **The entire byte is
zeroed on every save load.** Written `$FF`, booted the real ROM, read back **`0x00`**.

So these can *never* survive a save, and every one of them gets the **!**:

| bit | Where we keep it | Real name |
|---|---|---|
| 0 | `AreaNPC::tradeCenterSpritesFaced` | `BIT_INIT_TRADE_CENTER_FACING` |
| 3 | `AreaWarps::scriptedWarp` | `BIT_WARP_FROM_CUR_SCRIPT` |
| 4 | `AreaWarps::isDungeonWarp` | `BIT_ON_DUNGEON_WARP` |
| 5 | `AreaNPC::npcsFaceAway` | `BIT_NO_NPC_FACE_PLAYER` |
| 6 | `AreaPlayer::isBattle` | **`BIT_TALKED_TO_TRAINER`** |
| 7 | `AreaPlayer::isTrainerBattle` | **`BIT_PRINT_END_BATTLE_TEXT`** |

Every *other* warp byte we tampered with came back **exactly as written**. So the `!` list is short and
certain — which is the whole point of asking the console rather than guessing.

### ❌ Two of v1's fields are dead

`wWarpedFromWhichWarp` and `wWarpedFromWhichMap` — v1's *"From Warp"* and *"From Map"*. `WarpFound2`
writes them on every warp. **Nothing in the entire game reads either one.** Two writes, zero reads,
across the whole disassembly. They survive a save perfectly; they are simply **inert** — a breadcrumb
the game drops and never picks up.

⚠️ **A wiped byte and an unread byte are not the same fact**, and the panel must say which is which.

### …and the two that actually matter weren't on the warps screen at all

Twilight's instinct — *"from map needs to be in the top toolbar, it's pretty important to workflow"* —
is right; the byte named that is the dead one. The field she is reaching for is **`wLastMap`**
(`0x2611`): **the map that a `$FF` door returns you to.** Every building's exit is `$FF`. Walk out of
Red's house and you are in Pallet Town *because this byte says Pallet Town*. Its partner is
**`wLastBlackoutMap`** (`0x29C5`) — where blacking out, **Dig** and **Escape Rope** put you.

**Both are already modelled** (`WorldGeneral::lastMap` / `lastBlackoutMap`) — no new C++. They just have
to be *visible to the person editing warps*. So `wLastMap` goes in the **top toolbar**, in words —
**`[ Outside is: Pallet Town ⌄ ]`** — because changing it re-labels every `$FF` door on the map at once.

### Two hazards — and we are currently writing them ourselves

Same class as the music bank: a save value the console will use to read arbitrary ROM.

- **`wDestinationMap`** (Fly / special warp) is looked up in `FlyWarpDataPtr` — **13 entries, no
  terminator, no bounds check.** Any other map id runs the loop off the end of the table and copies six
  arbitrary ROM bytes into the view pointer and the player's coordinates.
- **`wDungeonWarpDestinationMap` + `wWhichDungeonWarp`** must be one of **12 legal (map, hole) pairs**,
  and the hole number is **1-based**. Victory Road 2F has no hole 1.

**And `AreaWarps::setTo()` / `randomize()` fill both with random illegal values today** — any cave, any
map, a 0-based hole index. Dormant only because `MapsDB` is never deep-linked at boot; **it goes live the
moment that landmine is defused.** Fixed in Phase 5a, exactly as `AreaAudio::setTo()` was fixed before
*its* path went live.

### The other model bugs found (seven, all in Phase 5a)

`blackoutDest` is not a destination — it is **`BIT_ESCAPE_WARP`** (Dig / Escape Rope / blackout), and it
is sitting in `AreaMap` as a bool doc'd *"may be unused"*. `skipJoypadCheckWarps` is **`BIT_FORCED_WARP`**
(doors fire without walking into them — it's how the Seafoam current sweeps you along). And `destWarp`
indexes the destination map's **`warps_to` arrival list**, not its warp list — the thing everyone gets
wrong.

### The screenshot review caught a cry-wolf, and it would have shipped

The fixture save — an ordinary one — holds `dungeonWarpDestMap = 194` and `whichDungeonWarp = 0`. That
pair isn't in `DungeonWarpList`, so the first cut lit a red **!** on it.

But **0 is the resting value**: `IsPlayerOnDungeonWarp` writes it as its *first instruction* whenever
you're not on a hole. **Every save ever made carries one**, `BIT_DUNGEON_WARP` is off, and the console
never looks. The warning was true and useless — it would have fired on every file anybody ever opened.
Precisely the mistake the sprite "your cast has changed" notice made first time round.

**The fix:** `legal` (is it in the table?) and `armed` (will the console read it?) are **two different
questions**, and the red **!** fires only on both. Out-of-table-but-inert gets a quiet grey line. And
the map and the hole are judged **separately** — the first cut failed both whenever the *pair* was
wrong, so a perfectly good hole map came up flagged because of the 0 beside it.

Pinned by `guns_dontCryWolfOnAnOrdinarySave`.
