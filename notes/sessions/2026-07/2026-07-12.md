## s15j — Music, Phases 3 & 4: the editor has a Game Boy in it now

A new library, **`pse-audio`**, with exactly two things in it:

- **`GbApu`** — the DMG sound chip. Four channels, the 512 Hz frame sequencer, envelopes, sweep (with the
  trigger-time overflow check), the LFSR, wave RAM, NR50/NR51 mixing, DAC-off-is-dead, and a box-averaged
  downsample to 48 kHz. It has never heard of Pokémon.
- **`Gen1SoundEngine`** — `engine_1.asm`, transliterated. Its whole state is a 256-byte array laid out
  exactly like the console's `$C000`, which is what will make the parity test possible.

The seam between them is the design: **the engine emits register writes and nothing else; the APU knows
nothing about notes.** Every hard question in Game Boy audio is a question about the chip, and every hard
question about the *music* is a question about the sequencer. Fusing them is how chiptune ports end up
sounding almost right and being unfixably wrong.

### `tst_audio` — 10 cases, green

The two that matter:

- **Pallet Town plays on channels 1, 2 and 3**, the pitches move (a tune moves; a stuck channel doesn't),
  and **the noise channel stays silent** — because Pallet Town has no drums.
- **Id 187 plays Pallet Town's channel 2 ALONE.** `wChannelSoundIDs = [0, 187, 0, 0]`, one hardware
  channel enabled, the melody moving. That is byte-for-byte what the cartridge did when we probed it. The
  "151 tracks for the price of 46" claim is no longer a claim.

Also pinned: DAC-off really kills a channel; NR11 reads its length bits back as 1s (not a curiosity — the
engine's duty-rotation does `and $3f` on that read-back, which is *why* rotating the duty writes a sound
length of 63); id 255 is silence.

### Everything kept, nothing "fixed"

The pitch table's signed negatives. The `and $c7` that clears length-enable, so the sound lengths the
engine writes are ignored and notes end because the *engine* starts the next one. The pitch-slide borrow
bug (pokered marks it; we keep it). The dead branch in perfect-pitch. Drums that are really *SFX calls*.
SFX channels that **pre-empt** the music rather than mix with it — the music channel keeps counting and
just stops writing registers, so it resumes exactly where it would have been.

### ⬜ Still owed before this can be called *right*

**`tst_sound_parity`.** Boot the real ROM in PyBoy, play track *N*, dump `$C000–$C0FF` **every frame**,
and demand our engine's 256 bytes match **frame by frame, for all 46 tracks**. Until that exists, this is
"sounds right", not "is right" — and this project does not accept the difference.

## s15i — Music, Phase 2: the data is in, and the cartridge signed off on every byte

`scripts/import_music.py` — the audio equivalent of `import_map_blocks.ps1`. It assembles the music
straight out of `pret/pokered`'s `.asm` (a small rgbasm subset: the whole `$10`–`$FF` macro set, local-label
scoping, and the `IF DEF(_RED)` branches — we are the **US English Red** cartridge), relocates every pointer
into our own address space, and writes:

```
projects/db/assets/data/music/bank02.bin   13,073 bytes
                              bank08.bin   11,927
                              bank1f.bin   13,520
                              waves.bin       128   (the 10 channel-3 wave instruments)
                              index.json
```

**38 KB, and it is the whole soundtrack** — every track, every sound effect stream, and (because the header
table is reproduced exactly) **all 105 inner voices for free**.

### It proves itself twice

1. **The ids.** A track's id is not stored anywhere — it is *computed from its header's address*. So the
   table's layout is load-bearing: one entry out of place and every id in the game shifts. The importer
   therefore **recomputes all 46 real track ids from the table it just built and demands they equal
   `music.json`'s.** One disagreement and it writes nothing.
2. **The bytes.** With the cartridge present it walks **our stream and the ROM's stream in lockstep**, every
   command, every operand — and for the pointer operands (which *must* differ, being at different addresses)
   it requires each ROM address to map to exactly one of ours. That's stronger than byte equality: it says
   the two are **the same graph**.

**All three banks: `MATCHES the cartridge`.**

### 🐞 And it immediately earned its keep

The lockstep check failed on the first run — and the bug was in *my* disassembler, in both this script and
the glitch-id analyser:

1. **`$2x` is only an SFX note on CHAN4–CHAN8.** On the three music tone channels it's an ordinary one-byte
   **note**. Reading it as a 4-byte `square_note` walks off the beat, and everything after it is nonsense.
2. **`sound_loop 0` is a terminator** — it never falls through. Walking past it wanders straight into the
   *next song's* bytes.

Both fixed; the glitch-id note counts in [`reference/glitch-music.md`](../../reference/glitch-music.md) are
corrected (the *structure* — which ids are inner voices, and of what — was never affected; it comes from the
header table, not the walk). **This is exactly what the cartridge cross-check is for: it caught a bug that
would have been inaudible-until-it-wasn't, and it caught it before a line of the engine was written.**

### One quirk imported verbatim, not fixed

**`.wave5` has no data.** The label exists; the bytes behind it are whatever happens to follow the wave table
in that bank — *different in all three*. Lavender Town and Pokémon Tower are built on the result. pokered
records exactly what those bytes are, and we reproduce them, per bank. It is not a bug to us; it is the
sound.

## s15h — Glitch music: there isn't any. There are 105 extra songs.

**The ask (Twilight).** *"Research what happens if glitch tracks are selected — since you have an audio
engine planned, follow a glitch track as far as you can, record the notes into sheet music, then we can
have more audio. A perk of a completely accurate sound emulation."*

She was right that there was a perk here. She was not right about what it was — it's **much better**.

Full write-up: [`reference/glitch-music.md`](../../reference/glitch-music.md).

### The finding

`music_constants.asm` opens with a line that turns out to be the whole story: *"Song ids are calculated by
address to save space."*

```
header address = SFX_Headers_N + id * 3
```

And **a header is 3 bytes per channel**. So a **3-channel song occupies three consecutive ids** — and only
the first is the song. The other two are its **channel 2 and channel 3 entries**, which parse *perfectly*
as headers in their own right (the channel-count bits are zero on a non-first entry, so they read as
one-channel headers) pointing at that song's channel-2 / channel-3 command stream.

**Id 186 is Pallet Town. Id 187 is Pallet Town's bassline, alone.**

So I parsed all **256 ids × 3 banks straight out of the cartridge** (`scripts/emu/analyze_music_ids.py` —
it does what `PlaySound` does, then disassembles every channel's command stream and follows its calls and
loops):

| Bank | Music region | Real tracks | **Inner voices** | Garbage |
|------|--------------|------------:|-----------------:|--------:|
| 2 | 186–254 | 20 | **49** | **0** |
| 8 | 234–254 | 7 | **14** | **0** |
| 31 | 195–254 | 19 | **42** | **0** |

**In the entire music region there is not one garbage id.** 105 inner voices. Lavender's channel 2 is a
**four-note drone that loops forever**. The Title Screen comes apart into four separate pieces. Gym Leader
Battle's channel 3 is 251 notes on its own.

### Then I made the console say it

`scripts/emu/probe_glitch_music.py` patches the id into a real save (**fixing the checksum**, or the game
refuses it), boots the cartridge, walks onto the map and reads the machine's own audio RAM and APU back:

| id | `wChannelSoundIDs` | NR51 (what reaches the speakers) |
|----|--------------------|----------------------------------|
| 186 (real) | `[186, 186, 186, 0…]` | `$77` — channels 1+2+3 |
| **187 ("glitch")** | `[0, **187**, 0, 0…]` | `$22` — **channel 2 only**, pitch moving through 10 values |

One channel loaded, one channel enabled, the melody moving. **That is not a glitch. That is a solo.**

Also confirmed on the console: an **SFX id** as map music plays the effect **once**, then silence; **id
255** is silence by design; and **id 0** — whose header is a `$FF $FF $FF` dummy — never reads that header
at all: `PlaySound` bails on a zero sound id and leaves the fade counter armed, so `FadeOutAudio` fades the
music out and then plays sound id **10** — a **drum hit**. The console agrees exactly: the only channel
that ever reaches the speakers with id 0 is NR51 `$88`, **the noise channel, alone**.

### ⚠️ And the one that bites

The **bank** byte is a different animal. `PlaySound` maps whatever bank the save names and *then* decides
which engine to call — so a bank that isn't 2/8/31 makes it call `Audio3_PlaySound`'s **address** with the
wrong bank mapped in: **arbitrary cartridge bytes, executed as code, every frame.**

**Verified.** With `bank = 5` the emulator ran the front end happily (720 frames in 0.1 s) and then, the
instant the save loaded and the map came up, **stopped producing frames entirely.** The console is dead.

So Phase 1 gains a guard rail: the bank picker offers **only 2 / 8 / 31**, and a save that already holds
something else is **shown, never silently rewritten** (fidelity is sacred), with a plain warning that it
hangs the real game.

### The perk, for real

Every inner voice points *into* a command stream we were already going to import. **Nothing in the music
region needs a byte of ROM we weren't already taking.** So the shipped editor — with no cartridge anywhere
near it — can play **46 real tracks + 105 inner voices = 151 pieces of audio, for the price of 46.**

And the sheet music she asked for is nearly free: because we run the **engine**, we *know* every note
exactly — pitch, octave, length, duty, volume, tempo — with no pitch detection and no guessing. A
`Transcriber` sink can export any of the 151 as MIDI and as real notation, following each track to its end
or its loop point (which the analyser already detects: `sound_loop` with count 0 *is* the repeat mark).
That's Phase 7 in [`plans/music.md`](../../plans/music.md).

## s15g — Music: the research pass (no code)

**The ask (Twilight).** The Map screen needs the **No Audio Fadeout** and **Prevent Music Change** flags —

Three decisions taken with her before a line was written:

1. **The music data is imported from `pret/pokered`** — the same precedent as the map `.blk`/`.bst` blocks.
   Music ships with the app; no ROM needed by anyone.
2. **Full port** — the real Gen 1 sequencer driving a real **DMG APU** model, in C++. Not an impression.
3. **This stretch is research + notes + plan only.** Nothing built yet.

### What was written

- **[`reference/gameboy-apu.md`](../../reference/gameboy-apu.md)** — the sound chip: 4 channels, every
  register, the frame sequencer, the frequency maths, the LFSR, DAC-off semantics, trigger semantics, and
  the `GbApu` shape we'll build.
- **[`reference/gen1-sound-engine.md`](../../reference/gen1-sound-engine.md)** — the game's sequencer: the
  full `$10`–`$FF` command set, the header format, the banks, the data formats, the quirks, and the two save
  flags.
- **[`plans/music.md`](../../plans/music.md)** — the six-phase plan, the architecture (`pse-audio`), the
  risks, and the open design questions for her.

### The five things that would have been got wrong

1. **There are not three engines.** `engine_1.asm` and `engine_3.asm` are **byte-for-byte identical** apart
   from a comment; `engine_2.asm` is the same engine plus the low-health alarm and the battle-cry modifiers.
   One C++ engine with a "bank 8" flag covers all three. Writing three would have been three times the bugs.
2. **The pitch table is negative numbers.** `$F82C` isn't a frequency, it's **−2004**. The engine shifts it
   right by `octave − 1` and then "adds 8 to the high byte" — which is **`+$800` = +2048**, reconstructing
   the register value. One 12-entry table covers eight octaves. Read it as unsigned and every note is wrong.
3. **Note length-enable is OFF.** The engine writes a sound length into `NRx1`, then `or $80` / `and $c7`
   **clears the length-enable bit**. The hardware ignores the length; notes end because the *engine* starts
   the next one. "Fixing" this would cut every note short.
4. **Drums are SFX.** A `drum_note` on the music noise channel *calls `PlaySound`* with a noise-instrument
   id. And SFX don't mix with music — they **pre-empt** it: the music channel keeps counting and just stops
   writing registers, so it resumes exactly where it would have been.
5. **The engine's entire mind is 243 bytes at `$C000`.** Which hands us the oracle: PyBoy can dump
   `$C000–$C0FF` from the real cartridge **every frame**, and our port must match it **byte-for-byte, frame
   by frame**. That is this work's `viewPointer_matchesWhatTheGameStored`.

### The two flags — verified, not guessed

The save→WRAM map for the main block is `file = 0x25A3 + (wram − 0xD2F7)`, cross-checked against the byte we
already knew (`0x2609` ⇒ `$D35D` = `wMapPalOffset`, the contrast byte). So:

- **`0x29D8` bit 1** = `$D72C` `wStatusFlags2` bit 1 = **`BIT_NO_AUDIO_FADE_OUT`**. What it *does*:
  `FadeOutAudio` normally slams the master volume back to full **every frame**; with this bit set it leaves
  the volume alone. The game sets it while the Pokédex and the status screen (the cry screens) are open.
- **`0x29DF` bit 1** = `$D733` `wStatusFlags7` bit 1 = **`BIT_NO_MAP_MUSIC`**. What it *does*: at the end of
  loading a map, the game skips starting the map's music — so whatever is playing keeps playing. It's why
  escort music follows you across map boundaries.

Our `AreaAudio` already reads and writes all four correctly.

### 🐞 One real bug found

`AreaAudio::setTo(MapDBEntry*)` does `musicBank = musicID = musicEntry->bank;` — it **clobbers the track id
with the bank**. Setting a map's default music writes the wrong track. Not fixed yet; it's Phase 1 of the
plan, with a test.

Twilight: *"figure out a place — I'm going to redo the UI/UX later anyway. Hover preview: nice. Map current
music should be selected by default."* So the placement is mine and **provisional**, which is itself the
design constraint: **the whole panel is one file**, so a future redesign moves one component and not a dozen

(with tooltips that say what they *actually* do, not what their names imply), and the 46 tracks **grouped**

**The rule for hover-preview: hover auditions, click commits.** It is armed only while playing (audio never
starts because a cursor drifted); a row must be settled on for ~120 ms, so sweeping the list doesn't

The map's current music is the **selected** track on open, so ▶ plays *this map's music* with no clicks. A

## s15f — The player is on the map, and the "harmless" glitch palettes stop being harmless

**The ask (Twilight).** *"Sprites is going to be another deep thing — you need to research the Game Boy and
the game code, pokered, and document all understanding in notes."* She's right, and it is.

Full write-up: [`reference/sprites.md`](../../reference/sprites.md) — the OAM hardware, Gen 1's 16-slot
overworld sprite system, and how the two meet.

### The three things that would have been got wrong

1. **There is no right-facing sprite.** `gfx/sprites/red.png` holds six frames — stand down, stand up, stand
   **left**, and three walking ones. Facing right is facing **left, X-flipped**
   (`SpriteFacingAndAnimationTable` → `.FlippedOAM`). Draw a "right" frame and you're inventing art the game
   doesn't have.
2. **There is no second walking frame either.** The 4-state walk cycle is `stand → walk → stand →
   walk-MIRRORED`. Gen 1 gets "steps with the other foot" out of **one drawing**, by flipping it.
3. **Sprites sit 4 pixels ABOVE their tile row** — *"which makes sprites appear to be in the centre of a
   tile"* (`ram/wram.asm`). Not folklore: **measured off the console's own OAM.** With the player at map
   (5,6), OAM entry 0 reads back as screen **(64, 60)** — and that is exactly our buffer position (176, 188)
   minus the screen origin (112, 128). Two independent routes, same pixel.

Also: the save's `playerCurDir` is **bit flags** (`1 right, 2 left, 4 down, 8 up`), *not* the sprite facing
values (`$0/$4/$8/$C`). Two encodings of the same idea, and mixing them up would quietly point him the wrong
way.

### The payoff — the glitch palettes finally bite

Yesterday's note said contrast **1 and 2 look completely normal on the map**, because they leave `rBGP` at
`$E4` and do their damage to `rOBP0`/`rOBP1` — the **object** palettes. There was nothing on screen to show
it.

**Now there is.** The player is drawn through `rOBP0`, so at contrast 1 the map stays perfectly normal and
*the player alone* goes wrong. That is exactly what a Game Boy does with that byte, and it is pinned:
`player_spriteIsCutOutAndPalettedLikeAnObject` asserts `backgroundPalette(1) == backgroundPalette(0)` **and**
`spritePalette(1) != spritePalette(0)` **and** that his sprite actually changes.

Colour 0 of an object is **always transparent** on the hardware — that's his cut-out. Without it he'd be a
white box.

### Verified

`tst_emu_parity` grew two cases and now demands the console agree about **where the player is** (its OAM) and
**which palettes it is holding** (`rBGP`/`rOBP0`). 9/9. `tst_map` 26/26.

**Not done yet, deliberately:** the NPCs (the other 15 slots, their sprite-set VRAM allocation, the `$FF`
off-screen rule, missables), the grass-priority bit (grass draws over a sprite's *lower half only*), and the
walk animation. All three are written up in `sprites.md` so the next step starts from knowledge, not a blank
page.

## s15e — The six glitch palettes are real, and they render

**The ask (Twilight).** *"You need to investigate what I call contrast — it's an array of entries that
relate to palette colors… on that list are also glitch palette colors. If you are rendering completely
accurately then the glitch palettes can actually render right, which would be awesome. There are 4 contrast
levels and 6 glitch palettes."* Plus: pinch-to-zoom and free horizontal + vertical scrolling.

**She was exactly right about the 4 and the 6 — and here is why.** Full write-up:
[`reference/palettes.md`](../../reference/palettes.md).

### What "contrast" is

The save byte at `0x2609` (`AreaGeneral::contrast`) is WRAM **`wMapPalOffset`**, and the game does something
gloriously unsafe with it (`home/fade.asm`):

```asm
LoadGBPal:  hl = FadePal4 - wMapPalOffset
            rBGP = [hl+] ; rOBP0 = [hl+] ; rOBP1 = [hl+]
```

It **subtracts the byte from a pointer into a table** of eight contiguous 3-byte fade palettes. Land on a
multiple of 3 and you get a real entry — `0, 3, 6, 9` → FadePal 4/3/2/1: **the four contrast levels**
(0 normal, 6 the "needs FLASH" cave palette, 9 black). Land anywhere else and the read **straddles two
entries**, producing three bytes that are not a palette anywhere in the game — `1, 2, 4, 5, 7, 8`: **the six
glitch palettes.** Exactly four and exactly six, and now we know *why*.

### Verified against the console

`scripts/emu/verify_palettes.py` writes each value into a save (checksum recomputed — a bad one is rejected
outright, so a broken save can never fake a pass), boots the real ROM, and reads the palette registers the
console actually ends up holding:

> **10 of 10. Zero mismatches.** Including all six glitch palettes.

Unlike the connection structs, this *is* verifiable through generated saves: `rBGP` is a hardware register,
recomputed live by `LoadGBPal` on every map load rather than restored from the save snapshot.

### Rendering them properly

The tileset art is four greys (`255/170/85/0`) captured under the identity palette — so **a pixel's shade
*is* its colour index**, and the remap is exact, not approximate:

```
index  = shade(pixel)
shade' = (rBGP >> (2 * index)) & 3
```

**We are not simulating "broken". We apply the byte the console would genuinely be holding** — which is why
the glitch palettes come out as the real article. Contrast 4 crushes the midtones in a way no legitimate
palette does; 6 is the near-black cave.

One honest subtlety: **contrast 1 and 2 look normal on the map**, because `rBGP` is still `E4` — their
damage is in `rOBP0`/`rOBP1`, the *sprite* palettes. They'll show the moment the player and NPCs are drawn.
That is not a gap in the render; it is what the console does.

### Navigation

Pinch-to-zoom (touch + touchpad) and Ctrl+wheel both route through one `zoomAround(newZoom, centre)` that
**anchors on what you're pointing at** — the cursor, or the pinch centroid — never the top-left corner.
Both scroll axes, drag-to-pan anywhere. It matters more the bigger the map gets, and it is about to get much
bigger.

### The screenshot review earned its keep again

The first cut put a Material `SpinBox` in the footer. It **shoved the zoom controls clean off the screen** —
Qt 6.5+ controls are large, the recurring "Material controls fight small heights" theme. Looked fine in the
code; visibly broken in the render. Contrast now lives in the info bar as a hand-rolled 22×22 `ContrastStep`
pair, and the value + its name both go red on a glitch palette rather than passing as an ordinary setting.

## s15d — Connection strips: understood in full, and the console agrees

**The ask (Twilight).** *"Connection strips are notoriously complicated and error prone. They have layers of
complexity, the algorithms are very complicated. I want you to very fully understand in detail them. The maps
need to be rendered extremely accurately, so accurately glitches are accurately rendered."* And, mid-session:
*"my formulas and algorithms could be wrong please verify accuracy."*

She was right on both counts. Full write-up: [`reference/map-connections.md`](../../reference/map-connections.md).

### The thing itself

A map's border ring is **not** a wall of the border block. The game fills it with the border block and then
**bleeds the neighbouring maps' edges over the top**. Pallet Town's ring is really *Route 1's bottom three
rows* and *Route 21's top three rows* — which is why you can see up the route before you walk there, and why
the walk across is seamless: you are already standing on its blocks.

The `connection` macro is a pure compile-time function of (direction, both maps' sizes, offset). Its layers:

- **The clamp.** `_tgt = offset + 3; if (_tgt < 2) { _src = -_tgt; _tgt = 0; }` — **two numbers out of one.**
  `_tgt` is where the strip lands in *our* ring; `_src` is how far into the *neighbour* it starts. A negative
  offset would push the destination out of the buffer, so the game slides the **source** along and pins the
  destination to 0. That asymmetry is the crux.
- **The length is limited by either map**: `min(curW + 3 - offset, toW)`, then minus `_src`.
- **The length byte means two different things.** North/south it is a **width** (3 rows of `length`);
  east/west it is a **row count** (`length` rows of 3). Same field, opposite meanings.
- **The source is a ROM POINTER**, not an array index — and the game reads it with `SwitchToMapRomBank` set
  to the **connected** map's bank. **66 of 78 connections are cross-bank.**

### Twilight's formulas — verified, and one is wrong

`MapDBEntryConnect::stripSize()` branches on `fromWidth < toWidth`. The macro clamps on
`min(curW + 3 - offset, toW)`. **Those are not the same test**, and `maps.json`'s **`flag`** field (21
connections carry it) is consulted *only* in the SOUTH/EAST branches of that function — it is a **patch for
that divergence**. The real game has no such concept. `MapEngine` therefore ignores both and recomputes from
the macro. (The DB still carries the wrong formula; fixing it touches curated data and a public API, so it's
**hers to call** — flagged, not done unasked.)

`maps.json` *does* carry everything needed, though: it stores the **post-clamp pair** (`stripMove == _tgt-3`,
`stripOffset == _src`), and the clamp is invertible —
**`offset = (stripOffset != 0) ? (-stripOffset - 3) : stripMove`**. Checked against the real headers: Route 4
south = **−25** (stored `-3, 22`), Route 11 east = **−27** (stored `-3, 24`), Viridian north = **+5**.

### The proof

`scripts/emu/verify_connections.py` finds **every map's header in the ROM** by its 10-byte signature, reads
the structs the macro compiled in, recomputes all 11 bytes from `maps.json`, and compares every field:

> **78 of 78 connections. Zero mismatches.**

And the ring it produces is byte-identical to the console's `wOverworldMap`
(`tst_emu_parity::theBorderRing_carriesTheConnectedMaps` — the `QEXPECT_FAIL` is **deleted**).

**A dead end worth remembering:** the first attempt drove the *emulator* — build a save placing the player on
each map, boot, read the structs from WRAM. It cannot work: **a Gen 1 save is a full WRAM snapshot**, storing
the live map header *and* the four connection structs, so the game never re-reads them on CONTINUE. Placing
the player elsewhere by save-editing means writing the very structs you are trying to verify. (It is also
*why* our `Area*` classes model all that live state.) The ROM has no such problem — it is what the game reads.

### Rendering glitches accurately

`BlocksDB::blockAt(bank, addr)` puts every map's blocks back where they live in the cartridge, so a strip
source is read as a **pointer**. No shipped connection reads past its neighbour's blocks (verified for all
78) — but a hand-edited or glitched offset can, and when it does it will overrun **exactly where the console
overruns**, into whatever map's blocks actually sit there. Addresses in ROM we don't hold return **−1**:
unknown, and said so, never invented.

### The ROM (the part that matters most)

Checked **before touching anything else**: `backup.gb` is git-ignored, **not tracked, and has never been in
the history** (checked against `git log --all`, not just the working tree). `.gitignore` now also refuses
`*.gb` / `*.gbc` / `*.gba` / `*.rom` **anywhere** in the tree, so it can't be added by accident even if it
moves. It is never published, released, or copied into a build artifact — only into git-ignored `tmp/emu/`.
Without it, every emulator case **SKIPs** and the suite stays green: nothing about building, testing or
shipping depends on it.

And its SHA-1 is **`ea9bcae6…` — exactly the `pokered.gbc` hash in the disassembly's own `roms.sha1`**. The
ROM, the disassembly we read the map format out of, and the editor are all the *same game*. That is what
makes the emulator a genuine oracle rather than an approximation.

### The harness

`scripts/emu/dump_state.py` drives **PyBoy** (headless): it boots the real ROM with one of our `.sav`
fixtures, lets the game load it, and dumps what the game *actually did* — `state.json` (including the view
pointer **it** computed), `wOverworldMap.bin`, `wSurroundingTiles.bin`, `wTileMap.bin`, and the 160×144
framebuffer. `tst_emu_parity` (CTest) then demands `MapEngine` match it **byte for byte**.

### The verdict

| The console's own RAM | Us |
|---|---|
| `wCurrentTileBlockMapViewPointer` | **MATCH** |
| the map's blocks in `wOverworldMap` | **MATCH** |
| `wSurroundingTiles` — the 24×20 scratch area | **MATCH** |
| `wTileMap` — the 20×18 tiles **on screen** | **MATCH** |
| the 3-block border ring | ❌ **differs** |

`wTileMap` matching is the big one: that array *is* the whole view pipeline — blocks → tiles → scratch area
→ half-block offset — with no sprites and no palettes in the way. The map emulator is right.

**And the border ring is genuinely wrong, exactly as suspected.** We fill it with the map's border block;
the game does that *and then bleeds the connected maps' edges over the top*. Pallet Town's ring is really
Route 1's bottom rows and Route 21's top rows — not a wall of trees. Connection strips are the next step,
and this is held as an explicit **`QEXPECT_FAIL`**, not a skip: the moment they land the test starts passing
and the expectation must be deleted, so the gap can't be quietly forgotten.

### Two traps, both worth remembering

- **The obvious "has the save loaded?" check is a trap.** "Does WRAM hold the save's map and coords?" passes
  **while the title screen is still up** — the main menu reads the save into WRAM just to decide whether to
  offer CONTINUE. It fooled this harness into a beautifully wrong parity failure. The emulator *screenshot*
  is what gave it away: WRAM said Pallet Town, the screen said "Pokémon — Red Version". The honest signal is
  **`wOverworldMap`**, which nothing fills but `LoadTileBlockMap`, and nothing calls that but `EnterMap`.
- **Bit 7 of the live `wCurMapTileset` is `BIT_NO_PREVIOUS_MAP`**, so Overworld reads back as 128, not 0.
  The save stores it clean — which is why the editor never sees it.

Also: **A is only pressed before the map loads.** Once on the overworld the harness touches no button at
all — A there would talk to a sign or an NPC and change the very state we came to measure.

## s15 — The map: scrapped, rebuilt, and now genuinely emulated

**The ask (Twilight).** The app has fully resurfaced from the revival; from here it's new territory.
Start with the biggest, most complicated part: **the map**. The old greyed-out Maps tile led to a menu
of dead ends — scrap it. Research `assets/references/pokered` ("the holy grail of all information this
editor uses") and **build a map emulator**, one step at a time. Step 1: take the **currently loaded map
and tileset**, render **all the tiles** out, then outline the **view screen** around the player — which
means understanding the viewbox properly, because there are several boxes (a draw area, a scratch area,
and the whole map). Render it *legitimately and exactly the way the game does*. No player sprite yet.

### What the game actually does (all of it read out of the disassembly)

The full, verified write-up now lives in [`reference/gen1-knowledge.md`](../../reference/gen1-knowledge.md)
→ "VERIFIED from the disassembly + real saves". The short version:

- A map is a grid of **blocks** (1 byte each, `maps/*.blk`). A block is **4×4 tiles = 32×32 px**, spelled
  out in the tileset's **blockset** (`gfx/blocksets/*.bst`, 16 bytes a block).
- `LoadTileBlockMap` doesn't keep the map on its own — it keeps it **ringed by a 3-block border**
  (`MAP_BORDER`), filled with the map's border block (which is literally the first byte of the map's
  object data = `maps.json`'s `border`). So the buffer is `(width+6) × (height+6)`.
- `LoadCurrentMapView` draws a **6×5-block scratch area** (24×20 tiles) out of that, and the **20×18-tile
  screen** out of *that*, offset by `(2·xBlockCoord, 2·yBlockCoord)` tiles. The scratch area is always
  block-aligned; the screen slides around inside it in half-block steps. **That is how the map scrolls
  smoothly while the grid it's built from never moves.**

### The proof it's an emulation, not an impression

The scratch area's top-left block is always the player's block **minus 2**, so:

```
viewPointer = 0xC6E8 + (1 + Y/2)·(width + 6) + (1 + X/2)
```

The save **stores the pointer the Game Boy itself computed** (`0x260B`). Recomputing it from nothing but
the player's coords and the map width reproduces it **byte-for-byte on both real saves** — Pallet Town
(10×9) and Red's House 2F (4×4). `tst_map::viewPointer_matchesWhatTheGameStored` pins it, and it's the
test that matters: if our idea of how the game builds its view ever drifts, that fails first.

It also explains **why the border is 3**: `(width+6) × (height+6)` is exactly the smallest buffer that
holds every view the player can produce, out to the corners. `view_everyViewFitsInsideTheBuffer` walks
every legal position on a real map and proves it.

### What got built

- **`scripts/import_map_blocks.ps1`** — imports the `.blk` + `.bst` data **verbatim** from the pokered
  clone into `db.qrc` (226 maps + 24 tilesets, ~50 KB), re-derivable and self-validating (`-Check`).
- **`BlocksDB`** (db) — serves the raw block data by the id the save stores.
- **`MapEngine`** (app) — rebuilds the overworld buffer and renders it at 1 screen px per Game Boy px;
  also carries the view maths. **`MapProvider`** serves it to QML as `image://map/<map>/<tileset>/<frame>`.
- **`MapModel`** (`brg.map`) — the loaded map + the three boxes, in buffer pixels.
- **`Map.qml`** — the new screen: the map inside its border ring, the block grid, and the three outlines.
  The old `Maps.qml` / `MapDetails.qml` / `MapSelectModel` / `IndividualMap` are **deleted**.
- **`tst_map`** — 18 cases. Full `ctest` **75/75**.

### Two data gaps found (Twilight's call — `maps.json` NOT touched)

The tests found both; neither is patched, because the JSON is curated data.

1. **22 glitch ids have no width/height** (11, 105–107, 109–112, 114–117, 204–206, 231, 237–238,
   241–244 — all `glitch: true`, "Unused Map XX"). The ROM is less tidy than our DB: its header table
   quietly points those ids at a **real map's header** (id 11 → Saffron City, 105–117 → Lance's Room,
   204–206 → Rocket Hideout Elevator, 231 → Route 16 Gate 1F, 237–244 → Silph Co 2F), so a Game Boy
   renders one. We don't import a map the DB can't size — the screen says so plainly instead. To make
   them render like the ROM, those entries need width/height/tileset (or an explicit alias field).
2. **3 sized "Copy" maps have an empty `tileset` string** — id 69 Trashed House Copy (ROM: House),
   75 Path Entrance Route 6 Copy (Gate), 173 Cinnabar Mart Copy (Mart). Harmless in-app (the loaded
   map's tileset comes from the **save**, which is correct); it only means the DB alone can't name it.
   Pinned by `tst_map` so the gap can't grow.

### A ROM quirk reproduced rather than fixed

`UNDERGROUND_PATH_NORTH_SOUTH` declares 4×24 = 96 blocks but its `.blk` is 92 bytes. The game copies
`width × height` regardless, so its last row is really the first 4 bytes of the *next* map's blocks in
ROM. The importer reads on through, exactly as the Game Boy does, and says so when it does.

### The QML landmine — the project's own note, ignored by its author

The block grid rendered **nothing**. No warning, `tst_qml_screens` green, and every other binding in the
file working perfectly. Cause: `Map.qml` was written with the habitual `id: top`, and **a Repeater
delegate cannot see the file's root id** — so `top.zoom` inside the delegate was `undefined`, `x` went
`NaN`, and Qt silently stacked all 17 grid lines at x = 0.

This is the *same landmine written up in [`reference/qt-patterns.md`](../../reference/qt-patterns.md)
yesterday* (the keyboard deck), and I walked straight into it. The note is now updated with the second,
nastier symptom — **NaN geometry doesn't vanish, it collapses to the origin**, so you get one plausible
artifact instead of an obvious hole — and with the technique that actually found it: colour the item
magenta, capture, and **pixel-sample the PNG**. (Plus the trap inside the trap: `Bitmap::FromFile` with
a relative path resolves against the .NET process cwd, not PowerShell's, and will hand you a confidently
wrong answer. It briefly convinced me a working fix hadn't worked.)

Root is now `id: mapScreen`. `required property int index` was suspected, investigated, and **cleared** —
it's fine.
