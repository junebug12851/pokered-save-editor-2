Twilight briefed a **Wild Pokémon** panel for the map screen's left dock: an enable button, an

**Research first (and it paid off).** `pret/pokered` `data/wild/maps/*` stores each slot as

cartridge-wrong bytes. Fixed it (a model phase before any UI), and pinned it with
`tst_area_pokemon::wildTables_byteOrderIsLevelThenSpecies` (level 7 + species 165 → disk `7` then
`165`). Also confirmed from the disassembly that `LoadWildData` is inside `LoadMapHeader` *after* the
`BIT_NO_PREVIOUS_MAP` early-return, so an edited table is **live on Continue** — the same linchpin as
warps/signs. All in the new `reference/wild-encounters.md`.

**Model surface.** `MapModel` gained `grassRate`/`waterRate`/`grassEnabled`/`waterEnabled`,
`grassMons()`/`waterMons()` (each slot → `{slot,index,level,name,dex,glitch,percent}`), per-slot

**UI.** `WildPokemonPanel.qml` (left dock) hosts two `WildMonList.qml` sections. Cells reuse

After the fixes the panel renders sprites, editable level chips ("L 11"…), percent and names cleanly.

**Also this session (Twilight's mid-turn correction):** she asked to **disable the Door meaning-layer

## Brief + build: the "3-step wild encounter cooldown" checkbox (map details page)

Twilight briefed **one checkbox** for the map details page — *"pause encounter for 3 steps"* — and
asked for research (it might be named/described/implemented wrong), a persistence test, and placement.

**Research → it's `BIT_WILD_ENCOUNTER_COOLDOWN`.** v2's `AreaPokemon::pauseMons3Steps` (save `0x29D8`
bit 0) is `wStatusFlags2` bit 0 (`$D72C`). The console **sets it after every battle** (`end_of_battle.asm`);
on the next map entry, if set, `EnterMap` loads `wNumberOfNoRandomBattleStepsLeft = 3` ("minimum number of
steps between battles") and each step decrements it until the bit self-clears. So it's a **post-battle
cooldown**, not a standalone "pause" toggle — but its save-editing *effect* is exactly "3 encounter-free
steps on the next Continue". New note: [`../../reference/wild-encounter-cooldown.md`](../../reference/wild-encounter-cooldown.md).
Same byte as the audio-fade flag (bit 1) — no collision (bit 0 vs bit 1).

**Console-verified (durable).** `scripts/emu/probe_wild_encounter_cooldown.py` stamps the bit, Continues on
the real ROM, reads it back **before any movement**: bit kept, `wNumberOfNoRandomBattleStepsLeft == 3`. The
**cleared control** reads steps-left 0 — proving the effect is caused by this bit (and confirming the
`$D13C` address, which swings 3↔0 with it). ⚠️ The clean **`BaseSAV` already carries it set** (saved right
after a battle) — so "on" is normal, not an edit. Because it is **kept** on load, it wears **no yellow "!"**.

**Model fix (Twilight's call): renamed `pauseMons3Steps` → `wildEncounterCooldown`** + doc naming
`BIT_WILD_ENCOUNTER_COOLDOWN` (`areapokemon.{h,cpp}`, `tst_area_pokemon`; byte-identical output). Exposed on
`MapModel` (new `AreaPokemon*` ctor arg from `bridge.cpp`) as a `wildEncounterCooldown` property whose setter
writes **exactly that one bit**.

**UI.** A `MapSwitch` on the map details page (nothing-selected section of `DetailsPanel.qml`), labelled
**"3-step wild encounter cooldown"** (Twilight's wording) with the honest one-line note. Screenshot-reviewed
(`tmp/cooldown-details.png`) — clean, switch reads ON on BaseSAV as expected.

**Green:** `tst_area_pokemon` 5/5, `tst_qml_screens` 16/16, `tst_map` 27/27; kit build clean.
⏳ **Owed: Twilight's live pass** (toggling on the real build).

## Brief + design: CONNECTIONS (connecting routes) — graduated to Phase 7

Twilight briefed the last big object type on the Map screen: the four edge **connections**. The
strips are already *rendered* and cartridge-verified (78/78,
[`../../reference/map-connections.md`](../../reference/map-connections.md)); this was about what
*editing* one should mean — v1's "Map" page exposed the raw 11-byte struct (Map ID, UL Corner,
Strip Src/Dest/Width, X/Y Align) and it was error-prone. No building; brief + design pass.

**The load-bearing insight** (now in the reference note, new "Editing a connection — the human
model" section): a connection has only **two human inputs** — the **neighbour map** and one
**signed offset (−27…+27)** — and the **other nine bytes are macro-derived**. So editing is the
derived-byte doctrine again: set map + offset, recompute the nine (via `MapEngine::connectionOf`,
never the wrong `MapDBEntryConnect::stripSize()`). Also pinned: **rotation is not representable in
the save** (a north connection always reads the neighbour's bottom 3 rows in natural orientation) —
the meaningful equivalent is **re-homing to another edge**, which just moves one flag bit + one
11-byte slot. Save footprint documented: flags byte `0x261C` (bits E/W/S/N) + four 11-byte blocks
(N `0x261D`, S `0x2628`, W `0x2633`, E `0x263E`).

**Twilight's four design answers (2026-07-15):**
1. **Offset-driven + auto-derive**, but the Details panel **always shows the raw values** (read-only
   while synced, editable once desynced).
2. **The live connection is drawn as the full neighbour map**, bleeding off the edge, **draggable**
   along the shared edge — the drag *is* the offset.
3. **Both add-gestures** — click a ghost edge-arrow (then pick a map) **and** drag a map onto the arrow.
4. **Never auto-rotate**; offer an explicit **"attach to another edge"** option (re-home), since that
   *is* representable. Handles are **simple when synced** (just slide); **richer nodes appear only on
   break-sync** (resize strip width, nudge src/dst).

Written up as **Phase 7 — Connections** (7a model-made-true → 7b canvas → 7c inspector → 7d
sync-aware handles → 7e honesty/tests) in [`../../plans/map-screen.md`](../../plans/map-screen.md);
the §12b "NOT YET BRIEFED" Connections row struck like Signs was, and the tail phases renumbered
(+1: Encounters→8, Area State→9, Tileset→10, Tools→11, Motion→12, Verify→13, Notes→14, Simulate→15).
Mid-brief addition (Twilight): the offset drag should **snap to landmark positions** (offset 0 /
flush edges / centred), with a light magnetism and `Alt` to escape — recorded in Phase 7b/7d.

## Live-review fixes (0.39.5-alpha) + the persistence question, answered

Twilight drove the built feature and reported:
- **🐞 An added connection was a dead state** — the neighbour rendered but there was no box to grab and
  it never "activated on the map side." **Cause:** the interactive strip came from
  `MapEngine::connectionStrips()`, which walks the map's **shipped DB** connections, so a *save-added*
  connection had no strip. **Fix:** the strip rect is now computed from the **save's own** connection in
  `connectionEditList` (`stripX/Y/W/H` + `hasStrip`, via `connectionOf`); `MapConnection` reads that, not
  the DB. New test `everyConnectionHasAnInteractiveStrip` (tst_connections 17/17).
- **Neighbour maps now animate** (water/flowers) — the render passes `brg.map.frame`.
- **Smarter picker** (`connectionMapList(dir)`): ★ **Default** (ROM's real neighbour for this edge) on
  top; then maps that connect on their **opposite** edge (they abut), grouped under a heading naming that
  edge; then the rest — each row shows **W×H** in grey on the right.

**Her persistence question, answered (and it matters):** *is a connection rewritten on startup, or kept
from the sav?* **Kept.** A Gen 1 save is a full WRAM snapshot; on **Continue** the game restores the four
connection structs from the save and does **not** re-read them from ROM (the same `BIT_NO_PREVIOUS_MAP`
linchpin as warps/signs — `map-connections.md`, "dead end" note). It rebuilds them from ROM **only** when
the player leaves the map and walks back in. So an edit **is live on load**, reverts on re-entry — which

re-entry. Added a one-line explicit answer to `map-connections.md`.

## Research: the seven leftover "Map" page fields (AreaMap state) — verified on the cartridge

Twilight is bringing v1's "Map" page (under "Area") into the map-details panel (shown when
nothing is selected) and asked what the fields actually are — they were "probably titled,
described, and organised wrong," and she wanted to know **which are rewritten at boot.** No
building; this is the research/brief pass.

Mapped her seven labels to the v2 `AreaMap` model, researched each in `pret/pokered`, then
**probed the real ROM** because a source read has been wrong before. Full write-up:
[`../../reference/area-map-state.md`](../../reference/area-map-state.md). Probe:
`scripts/emu/probe_area_map_state.py` (Pallet Town fixture, ordinary Continue).

**The verdicts (and the probe caught two source-reads being wrong):**

- **Current Script → `wCurMapScript` (`0x2CE5`) — a working index, half of a pair.** **Console:
  kept.** Deeper dig (2026-07-15): `RunMapScript` → `<Map>_Script` reads its **own per-map
  `w<Map>CurScript`** (there are **~90**, the real durable per-map progress, a separate un-briefed
  block ~`$D5F5`–`$D65C`), so `0x2CE5` is the **shared working index** written by
  `ExecuteCurMapScriptInTable`. And that routine, when `BIT_USE_CUR_MAP_SCRIPT` is set, **uses
  `wCurMapScript` as the index** — so **Current Script + "Run cur map script instead" are one
  feature in two bytes** (set both → run that step next tick). Descriptive combo source = the **98
  `_ScriptPointers` tables** (`SCRIPT_<MAP>_<NAME>`), extractable additively into `maps.json` (own
  data pass, ask before touching JSON).
- **UL Corner of Cur View → `wCurrentTileBlockMapViewPointer` (`0x260B`) — derived, but TRUSTED
  on load.** ⚠️ Predicted "recomputed every load"; **wrong** — the recompute lives in
  `CheckMapConnections`/warp-arrival, off the Continue path, and `LoadMapHeader` bails early
  (`BIT_NO_PREVIOUS_MAP`, the warp linchpin). Wrote `0xFFFF` → the cartridge **drew on-screen
  garbage** (`tmp/emu-areamap/areamap.png`). Doctrine (clarified by Twilight, below): **synced to
  the coords by default**, with a power-user break-sync path — not "show a mismatch + manual Sync".
- **UL Corner BG Tilemap in VR → `wMapViewVRAMPointer` (`0x27D2`) — reset every load.** Console:
  wrote `0xFFFF`, read `0x9800`. Live render pointer, not real state.
- **Run cur map script instead → `BIT_USE_CUR_MAP_SCRIPT` (`0x29DF` b4) — an auto-trigger lever.**
  ⚠️ Predicted "cleared on load"; **wrong for a quiet map** — `ExecuteCurMapScriptInTable`
  consumes it, but only scripted/trainer maps route through it, so it survived on Pallet. Because
  it survives, a save can carry it set to **auto-run a chosen map script on load** (Twilight's
  point) — an editable feature paired with the script-step selector, not dead plumbing.
- **force bike ride → `BIT_ALWAYS_ON_BIKE` (`0x29DE` b5) — durable, editable.** Cycling-Road
  lock. Console: kept (oak_speech.asm even notes it wrongly persists into a new game).
- **card key door x/y → `wCardKeyDoorX/Y` (`0x29EC`/`0x29EB`) — scratch, zeroed every load.**
  `ClearVariablesOnEnterMap` zeroes both. Console: read back `0`.
- **to blackout dest → ghost.** It was `BIT_ESCAPE_WARP`, **already moved to
  `AreaWarps::escapeWarp` on 2026-07-14.** Do not re-add it to the Map panel. The real blackout
  destination is `wLastBlackoutMap` in `WorldGeneral` (already surfaced). See `warps.md` §3.

**So "rewritten at boot" =** `mapViewVRAMPointer` (→ `$9800`) and `cardKeyDoorX/Y` (→ 0).
**Trusted =** `viewPointer` (derived), `curMapScript`, `forceBikeRide`. **Transient =**
`curMapNextFrame`. No save-corruption bug and no loaded gun here — it's a naming/organisation
problem plus one derived-value discipline (bug list in the reference note §6).

**Doctrine clarified (Twilight, this session) — the derived-value rule was over-stated.** It is
**not** "SHOWN, never silently synced / press a Sync button." The balance is: **keep derived
values in sync by DEFAULT** (the easy path — a novice editing the Map must not break it by not
hand-editing a derived field they had no reason to touch), **and** always expose raw-byte editing
plus a deliberate **break-sync** for power users (a toggle, an alert when a different value is
typed, or **dragging the view box on the canvas**). Byte fidelity still holds — an edit writes
only the bytes its action implies. Updated the canonical statement in **`CLAUDE.md`** (Critical
Things) and **`plans/map-screen.md` → §9 "The doctrine"** + the 4d-ii keystone, and folded through
`area-map-state.md` §2/§4/§6, `status.md`, the probe docstring, and this log.

**Design direction captured:** scripts = a ComboBox with an actually descriptive list + a
"Something else…" link for raw/hack values; the view/corner fields intuitively selectable and
**never an address format** except behind a "Something else…" link. Folded into `area-map-state.md`
§2/§4 and `plans/map-screen.md` → Phase 8 (Area State).

**Files:** new `notes/reference/area-map-state.md`; new `scripts/emu/probe_area_map_state.py`;
wired into `_nav.dox`, `CLAUDE.md` notes table, `status.md`, `plans/map-screen.md`.
