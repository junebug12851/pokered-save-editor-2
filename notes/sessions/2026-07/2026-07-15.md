## New feature briefed: the Events screen ‚Äî the 2,560 `wEventFlags` bits (Phase 0‚Äì1 foundation)

Project leadership briefed a **colossal new feature**: research and properly edit **every one of the
2,560 world-event flags** (v1's cryptic "Events" page ‚Äî `1cc`, `1cd`, ‚Ä¶). The mandate: every single bit
‚Äî named, unnamed, unused, temporary, glitchy ‚Äî gets a **proper name, description, map** (or a well-named
general group, split if too big), **flag-group** membership + group toggles, and a **classification**
(unused / duplicate / multi-purpose / temporary / crash). Crash flags & combos must warn the user (panel
+ on-toggle). Exhaustive; no shortcuts for the unused/temporary. This feature **first**, then the
persistent-storage map panel.

**Verified the ground truth against `pret/pokered` (not memory):** `wEventFlags = flag_array NUM_EVENTS`,
`NUM_EVENTS = $A00 = 2560` bits = **320 bytes**, WRAM `0xD747`‚Äì`0xD886`, save-file **`0x29F3`‚Äì`0x2B32`**,
ending exactly where `wGrassRate` begins (`0x2B33`). So there are **not more than 2,560** ‚Äî the count is
fixed by the ROM.

**pret already names 507; 2,053 are gaps** ‚Äî but per the mandate all 2,560 get the full treatment.
`event_constants.asm` also groups the named ones by `; <City> events` blocks, so ~507 arrive with an
implied map.

**Phase 1 done ‚Äî the canonical bit-map.** `scripts/import_event_flags.py` parses
`const_def`/`const`/`const_skip`/`const_next` into all 2,560 rows (index, byte_wram, byte_file, bit, pret
name, owning map-region), **self-validating** to exactly `$A00`, and attributes **every** bit ‚Äî named and
gap ‚Äî to its map-region (pret allocates each city a contiguous block via `const_next`). Output:
`tmp/event-flags/event_flags_canonical.{csv,json}`. Coverage: **2,560 = 507 named + 2,053 gap across 46
map-regions**, every gap given a home (no orphans).

**Naming rule (leadership):** real discerned name wherever findable; **"Unknown #<hex id>"** is a
last-resort fallback **only** after exhausting every file **incl. the raw assembly**, and **only with
explicit leadership sign-off** (per-flag or per-group) ‚Äî never a silent default. "Unknown"
(undiscoverable) ‚âÝ "unused" (clear identity, no references).

**Also captured (standing rules):** address decisions as **"project leadership,"** not "ask Twilight";

## Brief + build: the "3-step wild encounter cooldown" checkbox (map details page)

Twilight briefed **one checkbox** for the map details page ‚Äî *"pause encounter for 3 steps"* ‚Äî and
asked for research (it might be named/described/implemented wrong), a persistence test, and placement.

**Research ‚Üí it's `BIT_WILD_ENCOUNTER_COOLDOWN`.** v2's `AreaPokemon::pauseMons3Steps` (save `0x29D8`
bit 0) is `wStatusFlags2` bit 0 (`$D72C`). The console **sets it after every battle** (`end_of_battle.asm`);
on the next map entry, if set, `EnterMap` loads `wNumberOfNoRandomBattleStepsLeft = 3` ("minimum number of
steps between battles") and each step decrements it until the bit self-clears. So it's a **post-battle
cooldown**, not a standalone "pause" toggle ‚Äî but its save-editing *effect* is exactly "3 encounter-free
steps on the next Continue". New note: [`../../reference/wild-encounter-cooldown.md`](../../reference/wild-encounter-cooldown.md).
Same byte as the audio-fade flag (bit 1) ‚Äî no collision (bit 0 vs bit 1).

**Console-verified (durable).** `scripts/emu/probe_wild_encounter_cooldown.py` stamps the bit, Continues on
the real ROM, reads it back **before any movement**: bit kept, `wNumberOfNoRandomBattleStepsLeft == 3`. The
**cleared control** reads steps-left 0 ‚Äî proving the effect is caused by this bit (and confirming the
`$D13C` address, which swings 3‚Üî0 with it). ‚öÝÔ∏è The clean **`BaseSAV` already carries it set** (saved right
after a battle) ‚Äî so "on" is normal, not an edit. Because it is **kept** on load, it wears **no yellow "!"**.

**Model fix (Twilight's call): renamed `pauseMons3Steps` ‚Üí `wildEncounterCooldown`** + doc naming
`BIT_WILD_ENCOUNTER_COOLDOWN` (`areapokemon.{h,cpp}`, `tst_area_pokemon`; byte-identical output). Exposed on

writes **exactly that one bit**.

**UI.** A `MapSwitch` on the map details page (nothing-selected section of `DetailsPanel.qml`), labelled
**"3-step wild encounter cooldown"** (Twilight's wording) with the honest one-line note. Screenshot-reviewed

## Dev-harness: universal by-name + by-path navigation, trigger-anything, component-cropped shots

Driving the new panel exposed a real harness gap (Twilight caught it): rail buttons *have*
`objectName`s (`dockBtn_charstate`) yet `list`/`click` couldn't find them, so I fell back to guessing
`tap` coordinates. Cause: `list`/`findItem` walked only the **QObject** tree (`findChildren`), which
misses `Repeater`/model delegates ‚Äî every rail button, list row, anything built from an array. Fixes,
all in `debugserver.cpp` + `mainwindow.{h,cpp}` (DEBUG-only), per her follow-ups:

- **Everything named is reachable** ‚Äî `list`/`findItem` now walk the QObject tree **and** the Qt Quick
  visual tree (`childItems()`). `list dockBtn` ‚Üí all seven rail buttons; `click dockBtn_charstate` opens
  the panel by name.
- **Reach even UNNAMED nodes, by PATH** ‚Äî any `obj` accepts a `/`-separated path from a named ancestor:
  a number indexes direct children (visual order), a word matches a child `objectName`. `list obj=<node>`
  enumerates a node's children (`[i] Type name`) so you can step down. Verified: `get mapCharacterState/0
  width` ‚Üí 240.
- **Trigger anything** ‚Äî `invoke` emits any signal / calls any slot/`Q_INVOKABLE` by name; `tap` gained
  `double` and `button` (right/middle) for double-clicks and context menus.
- **Screenshot from any component down** ‚Äî `shot obj=<node>` crops to that component's bounds (DPR-aware),

## Brief + design: CONNECTIONS (connecting routes) ‚Äî graduated to Phase 7

Twilight briefed the last big object type on the Map screen: the four edge **connections**. The
strips are already *rendered* and cartridge-verified (78/78,
[`../../reference/map-connections.md`](../../reference/map-connections.md)); this was about what
*editing* one should mean ‚Äî v1's "Map" page exposed the raw 11-byte struct (Map ID, UL Corner,
Strip Src/Dest/Width, X/Y Align) and it was error-prone. No building; brief + design pass.

**The load-bearing insight** (now in the reference note, new "Editing a connection ‚Äî the human
model" section): a connection has only **two human inputs** ‚Äî the **neighbour map** and one
**signed offset (‚àí27‚Ä¶+27)** ‚Äî and the **other nine bytes are macro-derived**. So editing is the
derived-byte doctrine again: set map + offset, recompute the nine (via `MapEngine::connectionOf`,
never the wrong `MapDBEntryConnect::stripSize()`). Also pinned: **rotation is not representable in
the save** (a north connection always reads the neighbour's bottom 3 rows in natural orientation) ‚Äî
the meaningful equivalent is **re-homing to another edge**, which just moves one flag bit + one
11-byte slot. Save footprint documented: flags byte `0x261C` (bits E/W/S/N) + four 11-byte blocks
(N `0x261D`, S `0x2628`, W `0x2633`, E `0x263E`).

**Twilight's four design answers (2026-07-15):**
1. **Offset-driven + auto-derive**, but the Details panel **always shows the raw values** (read-only
   while synced, editable once desynced).
2. **The live connection is drawn as the full neighbour map**, bleeding off the edge, **draggable**
   along the shared edge ‚Äî the drag *is* the offset.
3. **Both add-gestures** ‚Äî click a ghost edge-arrow (then pick a map) **and** drag a map onto the arrow.
4. **Never auto-rotate**; offer an explicit **"attach to another edge"** option (re-home), since that
   *is* representable. Handles are **simple when synced** (just slide); **richer nodes appear only on
   break-sync** (resize strip width, nudge src/dst).

Written up as **Phase 7 ‚Äî Connections** (7a model-made-true ‚Üí 7b canvas ‚Üí 7c inspector ‚Üí 7d
sync-aware handles ‚Üí 7e honesty/tests) in [`../../plans/map-screen.md`](../../plans/map-screen.md);
the ¬ß12b "NOT YET BRIEFED" Connections row struck like Signs was, and the tail phases renumbered
(+1: Encounters‚Üí8, Area State‚Üí9, Tileset‚Üí10, Tools‚Üí11, Motion‚Üí12, Verify‚Üí13, Notes‚Üí14, Simulate‚Üí15).
Mid-brief addition (Twilight): the offset drag should **snap to landmark positions** (offset 0 /
flush edges / centred), with a light magnetism and `Alt` to escape ‚Äî recorded in Phase 7b/7d.

Twilight drove the built feature and reported:

  edge; then the rest ‚Äî each row shows **W√óH** in grey on the right.

**Her persistence question, answered (and it matters):** *is a connection rewritten on startup, or kept
from the sav?* **Kept.** A Gen 1 save is a full WRAM snapshot; on **Continue** the game restores the four
connection structs from the save and does **not** re-read them from ROM (the same `BIT_NO_PREVIOUS_MAP`
linchpin as warps/signs ‚Äî `map-connections.md`, "dead end" note). It rebuilds them from ROM **only** when
the player leaves the map and walks back in. So an edit **is live on load**, reverts on re-entry ‚Äî which
is exactly why it's *not* marked "rewritten on startup" and why the honest note only warns about
re-entry. Added a one-line explicit answer to `map-connections.md`.

`buildOverworldMap`/`applyConnections` read the DB; threading the save's live connections through the
render pipeline is its own change. (2) **Living/wandering NPCs + player collision** (below).

## Research: the seven leftover "Map" page fields (AreaMap state) ‚Äî verified on the cartridge

Twilight is bringing v1's "Map" page (under "Area") into the map-details panel (shown when
nothing is selected) and asked what the fields actually are ‚Äî they were "probably titled,
described, and organised wrong," and she wanted to know **which are rewritten at boot.** No
building; this is the research/brief pass.

Mapped her seven labels to the v2 `AreaMap` model, researched each in `pret/pokered`, then
**probed the real ROM** because a source read has been wrong before. Full write-up:
[`../../reference/area-map-state.md`](../../reference/area-map-state.md). Probe:
`scripts/emu/probe_area_map_state.py` (Pallet Town fixture, ordinary Continue).

**The verdicts (and the probe caught two source-reads being wrong):**

- **Current Script ‚Üí `wCurMapScript` (`0x2CE5`) ‚Äî a working index, half of a pair.** **Console:
  kept.** Deeper dig (2026-07-15): `RunMapScript` ‚Üí `<Map>_Script` reads its **own per-map
  `w<Map>CurScript`** (there are **~90**, the real durable per-map progress, a separate un-briefed
  block ~`$D5F5`‚Äì`$D65C`), so `0x2CE5` is the **shared working index** written by
  `ExecuteCurMapScriptInTable`. And that routine, when `BIT_USE_CUR_MAP_SCRIPT` is set, **uses
  `wCurMapScript` as the index** ‚Äî so **Current Script + "Run cur map script instead" are one
  feature in two bytes** (set both ‚Üí run that step next tick). Descriptive combo source = the **98
  `_ScriptPointers` tables** (`SCRIPT_<MAP>_<NAME>`), extractable additively into `maps.json` (own
  data pass, ask before touching JSON).
- **UL Corner of Cur View ‚Üí `wCurrentTileBlockMapViewPointer` (`0x260B`) ‚Äî derived, but TRUSTED
  on load.** ‚öÝÔ∏è Predicted "recomputed every load"; **wrong** ‚Äî the recompute lives in
  `CheckMapConnections`/warp-arrival, off the Continue path, and `LoadMapHeader` bails early
  (`BIT_NO_PREVIOUS_MAP`, the warp linchpin). Wrote `0xFFFF` ‚Üí the cartridge **drew on-screen
  garbage** (`tmp/emu-areamap/areamap.png`). Doctrine (clarified by Twilight, below): **synced to
  the coords by default**, with a power-user break-sync path ‚Äî not "show a mismatch + manual Sync".
- **UL Corner BG Tilemap in VR ‚Üí `wMapViewVRAMPointer` (`0x27D2`) ‚Äî reset every load.** Console:
  wrote `0xFFFF`, read `0x9800`. Live render pointer, not real state.
- **Run cur map script instead ‚Üí `BIT_USE_CUR_MAP_SCRIPT` (`0x29DF` b4) ‚Äî an auto-trigger lever.**
  ‚öÝÔ∏è Predicted "cleared on load"; **wrong for a quiet map** ‚Äî `ExecuteCurMapScriptInTable`
  consumes it, but only scripted/trainer maps route through it, so it survived on Pallet. Because
  it survives, a save can carry it set to **auto-run a chosen map script on load** (Twilight's
  point) ‚Äî an editable feature paired with the script-step selector, not dead plumbing.
- **force bike ride ‚Üí `BIT_ALWAYS_ON_BIKE` (`0x29DE` b5) ‚Äî durable, editable.** Cycling-Road
  lock. Console: kept (oak_speech.asm even notes it wrongly persists into a new game).
- **card key door x/y ‚Üí `wCardKeyDoorX/Y` (`0x29EC`/`0x29EB`) ‚Äî scratch, zeroed every load.**
  `ClearVariablesOnEnterMap` zeroes both. Console: read back `0`.
- **to blackout dest ‚Üí ghost.** It was `BIT_ESCAPE_WARP`, **already moved to
  `AreaWarps::escapeWarp` on 2026-07-14.** Do not re-add it to the Map panel. The real blackout
  destination is `wLastBlackoutMap` in `WorldGeneral` (already surfaced). See `warps.md` ¬ß3.

**So "rewritten at boot" =** `mapViewVRAMPointer` (‚Üí `$9800`) and `cardKeyDoorX/Y` (‚Üí 0).
**Trusted =** `viewPointer` (derived), `curMapScript`, `forceBikeRide`. **Transient =**
`curMapNextFrame`. No save-corruption bug and no loaded gun here ‚Äî it's a naming/organisation
problem plus one derived-value discipline (bug list in the reference note ¬ß6).

**Doctrine clarified (Twilight, this session) ‚Äî the derived-value rule was over-stated.** It is
**not** "SHOWN, never silently synced / press a Sync button." The balance is: **keep derived
values in sync by DEFAULT** (the easy path ‚Äî a novice editing the Map must not break it by not
hand-editing a derived field they had no reason to touch), **and** always expose raw-byte editing
plus a deliberate **break-sync** for power users (a toggle, an alert when a different value is
typed, or **dragging the view box on the canvas**). Byte fidelity still holds ‚Äî an edit writes
only the bytes its action implies. Updated the canonical statement in **`CLAUDE.md`** (Critical
Things) and **`plans/map-screen.md` ‚Üí ¬ß9 "The doctrine"** + the 4d-ii keystone, and folded through
`area-map-state.md` ¬ß2/¬ß4/¬ß6, `status.md`, the probe docstring, and this log.

**Design direction captured:** scripts = a ComboBox with an actually descriptive list + a
"Something else‚Ä¶" link for raw/hack values; the view/corner fields intuitively selectable and
**never an address format** except behind a "Something else‚Ä¶" link. Folded into `area-map-state.md`
¬ß2/¬ß4 and `plans/map-screen.md` ‚Üí Phase 8 (Area State).

**Files:** new `notes/reference/area-map-state.md`; new `scripts/emu/probe_area_map_state.py`;
