And the three were the **exact opposite** of "the container lacks the ROM + PyBoy":

```
FAIL!  : TstSoundParity::engineMatchesTheConsole(TrainerBattle) 'rc == 0' returned FALSE.
         (the emulator could not play this track:
          <3>WSL (3483 - ) ERROR: UtilGetPpid:1330: Failed to parse: /proc/1/stat ‚Ä¶)
```

`docker/run-tests.sh` rsyncs the repo into the container excluding only `.git/` and the build dirs ‚Äî
so the host's `tmp/` rode along, **Windows `emu-venv` and all**. The skip-gate was
`QFile::exists("tmp/emu-venv/Scripts/python.exe")`, hardcoded to the Windows layout in **three
copies**. On Linux that file *was there*, so the gate answered **available**, the QSKIP never fired,
and the test tried to `exec` a **Windows PE binary**. WSL's binfmt interop grabbed it and failed
sideways ‚Äî which surfaced as red tests that looked like a code fault and weren't. **CI was green
precisely because a fresh checkout has no `tmp/`.**

‚≠ê **The lesson, and it is bigger than this bug:** *a file existing is not a capability being
available.* An availability gate that tests for a **path** instead of the **capability** will pass in
exactly the environment it was written to protect. It held only because the one machine that had a
venv was the one that made it.

**Fixed twice, deliberately** ‚Äî belt *and* braces, because either alone leaves the trap set:

- **The gate is now runnability-based**: new shared `projects/tests/helpers/emuvenv.h`, used by all
  three tests. Platform-correct path (`Scripts/python.exe` ‚Üî `bin/python3`) and then the only
  question that matters ‚Äî **`pythonRuns()`**: spawn `--version`, demand exit 0, cache it.
- **`run-tests.sh` excludes `tmp/`** ‚Äî host scratch (screenshots, emu output, venv) has no business
  in the container regardless.

**Proven independently**, because otherwise the exclusion would hide whether the gate works. Re-injected
the Windows venv into the container by hand and ran the test:

```
--- windows venv IS present in container ---
SKIP   : TestEmuParity::initTestCase() emulator verification unavailable: tmp/emu-venv holds an
         interpreter that will not run on this platform (a venv built for another OS)
         -- rebuild it with scripts/emu/setup.ps1
```

A skip that **explains itself**, where there used to be WSL garbage.

**Green:** container **91/91 standard**, **91/91 asan**, **0 sanitizer findings, 0 fontconfig**.
Windows kit: the three emu tests still **pass for real** (the venv runs here) ‚Äî the gate had to keep
working in the positive case, not just the negative one.

### 1. A heap-use-after-free in the music parser (the serious one)

`Gen1MusicAsm::parse` ‚Üí `addrOf()` took **`const QString& label`**, and the caller passes a reference
**straight into `pending`**:

```cpp
auto addrOf = [&](const QString& label) -> int {   // <- reference INTO `pending`
  ...
  if (!placed.contains(file))
    place(file);                                   // <- place() does pending.append(...)  ==> REALLOC
  return placed[file] + streams[file].labels.value(label);   // <- `label` now dangles. READ of freed.
};

for (int i = 0; i < pending.size(); i++)           // the loop's own comment: "`pending` GROWS"
  const int addr = addrOf(pending.at(i).second);
```

The growth was known ‚Äî the comment says so. That the growth **frees the caller's reference** was not.
Taken **by value** now.

ASan reported it **7 times from 5 different tests** (`tst_audio`, `tst_music_asm`, `tst_acceptance`,
`tst_gui_fidelity`, `tst_qml_screens`) ‚Äî all **one root cause**: everything that parses the sheet music
went through it. It never crashed, because freed memory usually still holds what you left there. That
is exactly what makes the class dangerous.

### 2‚Äì4. The clang-tidy gate (3 findings, all legitimate)

- **`MapModel::mapInd()` ‚Äî null dereference.** Bare `map->curMap`. `map` **is** nullable and this class
  says so itself: the ctor null-checks every pointer it connects to, and `connectionRoomLeft()` /
  `saveConns()` both guard `map`. **That is precisely why the analyzer could prove the path** ‚Äî the
  code's own guards are the evidence. Now guarded, returning 0 like its siblings.
- **`mapengine.cpp:662` ‚Äî redundant expression, with a real bug hiding behind it.**
  `mapInd < FirstIndoorMapId && mapInd < NumCityMaps` ‚Äî `NumCityMaps` (10) is far below
  `FirstIndoorMapId` (0x25), so the second **always implied** the first. The check that was *missing*
  is **`>= 0`**, on an expression that indexes `SgbCity`. Dropped the dead half, added the real one.
- **A `Strip` copied where a `const&` would do.**

### The reason none of this was caught here ‚Äî BOTH safety nets were down

- **ASan does not run on the llvm-mingw kit.** The **container** is the only place it runs‚Ä¶

`libpulse-mainloop-glib0`** ‚Äî `libQt6Multimedia.so` carries a hard `DT_NEEDED` on `libpulse.so.0`, so
without it **every test that links audio fails at the link step**, though nothing here ever plays a
sound. GitHub's runner ships PulseAudio, which is why the workflows install no apt deps and never hit
it; a clean container must ask.
