### 1. A heap-use-after-free in the music parser (the serious one)

`Gen1MusicAsm::parse` ‚Üí `addrOf()` took **`const QString& label`**, and the caller passes a reference
**straight into `pending`**:

```cpp
auto addrOf = [&](const QString& label) -> int {   // <- reference INTO `pending`
  ...
  if (!placed.contains(file))
    place(file);                                   // <- place() does pending.append(...)  ==> REALLOC
  return placed[file] + streams[file].labels.value(label);   // <- `label` now dangles. READ of freed.
};

for (int i = 0; i < pending.size(); i++)           // the loop's own comment: "`pending` GROWS"
  const int addr = addrOf(pending.at(i).second);
```

The growth was known ‚Äî the comment says so. That the growth **frees the caller's reference** was not.
Taken **by value** now.

ASan reported it **7 times from 5 different tests** (`tst_audio`, `tst_music_asm`, `tst_acceptance`,
`tst_gui_fidelity`, `tst_qml_screens`) ‚Äî all **one root cause**: everything that parses the sheet music
went through it. It never crashed, because freed memory usually still holds what you left there. That
is exactly what makes the class dangerous.

### 2‚Äì4. The clang-tidy gate (3 findings, all legitimate)

- **`MapModel::mapInd()` ‚Äî null dereference.** Bare `map->curMap`. `map` **is** nullable and this class
  says so itself: the ctor null-checks every pointer it connects to, and `connectionRoomLeft()` /
  `saveConns()` both guard `map`. **That is precisely why the analyzer could prove the path** ‚Äî the
  code's own guards are the evidence. Now guarded, returning 0 like its siblings.
- **`mapengine.cpp:662` ‚Äî redundant expression, with a real bug hiding behind it.**
  `mapInd < FirstIndoorMapId && mapInd < NumCityMaps` ‚Äî `NumCityMaps` (10) is far below
  `FirstIndoorMapId` (0x25), so the second **always implied** the first. The check that was *missing*
  is **`>= 0`**, on an expression that indexes `SgbCity`. Dropped the dead half, added the real one.
- **A `Strip` copied where a `const&` would do.**

### The reason none of this was caught here ‚Äî BOTH safety nets were down

- **ASan does not run on the llvm-mingw kit.** The **container** is the only place it runs‚Ä¶
- **‚Ä¶and the container had been broken since `pse-audio` landed.** Its image installed Qt with
  `-m qtcharts` alone, so once the build needed Qt Multimedia every run died at **configure** ‚Äî
  *"Could NOT find Qt6Multimedia"* ‚Äî before compiling a line. `dtest.ps1 asan` had not run in a long
  time and said so only in a log nobody read.
- **The remote could not compile** (the 6.9-only `QImage::flipped`).

So the one local tool that would have caught the UAF **could not start**, and the remote that would
have caught it **could not build**. That is the whole story of how it survived twelve versions.

**Container fixed** (`b070f1a` + follow-up): `-m qtcharts qtmultimedia qtshadertools` (now matching
`tests.yml`'s `modules:` line, with a comment to keep them in step) plus **`libpulse0` /
`libpulse-mainloop-glib0`** ‚Äî `libQt6Multimedia.so` carries a hard `DT_NEEDED` on `libpulse.so.0`, so
without it **every test that links audio fails at the link step**, though nothing here ever plays a
sound. GitHub's runner ships PulseAudio, which is why the workflows install no apt deps and never hit
it; a clean container must ask.

**Verified:** local `dtest.ps1 asan` now builds and runs the suite again ‚Äî **0 ASan/UBSan findings** ‚Äî
and CI's `linux-asan` job is **success** on the same commit. Full local `ctest` **91/91**.
