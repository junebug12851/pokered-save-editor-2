# 2026-07-17

## 📏 THE UNITS: there are THREE, and we'd been calling the middle one a "tile"

Twilight, on the ranged-box unit question: *"arent some filter flags or coords or whatever for things
in meansurement of blocks, i like blocks as a measurement. Things like tile attributes would be
examples of tile measurements"* → then the rule that settles it: **"use the measurement its suppose to
be"**.

Checked against `pret/pokered` rather than agreed with, and it turned out **half right — with the
wrong half being the interesting one.**

| Unit | Size | What is measured in it |
|---|---|---|
| **Tile** | 8×8 | tile attributes — collision, ledges, water, counters, warp tiles. ✅ **her instinct, exactly right** |
| **Half-block** *(walk grid)* | **16×16 = 2×2 tiles** | `wYCoord`/`wXCoord` → **objects (→ filter flags) · warps · signs · hidden items + coins · script coord triggers** |
| **Block** | 32×32 = 4×4 tiles | the `.blk` data, map width/height, the border block, connections |

**The game proves the middle unit itself** (`engine/gfx/tilesets.asm`):

```asm
ld a, [wYCoord]
and $1                  ; the LOW BIT of the coord ...
ld [wYBlockCoord], a    ; ... IS "which half of the block am I in"
```

…and its consumer is named **`.adjustForYCoordWithinTileBlock`**. That `& 1` is nonsense unless
`wYCoord` counts **half-blocks**. Corroborated: `CheckWarpsNoCollision` loads `wYCoord` into `d` and
compares each warp's stored Y straight against it (same unit, no conversion); `macros/maps.asm` emits
`object_event` as `db \2 + 4, \1 + 4` and `bg_event` as `db \2, \1, \3` on that same grid.

⭐ **So nothing we box is block-scoped — and a block-sized box would be a LIE:** it spans 2×2 walk
squares and cannot tell two adjacent warps apart. Against *"use the measurement its suppose to be"*,
that settles it: **the 16×16 boxes already shipping are correct. The word was wrong, not the size.**

🐺 **And the bad vocabulary was ours.** `tiles.md` has said *"Gameplay happens at the **tile** level
(you walk in half-blocks = 2×2 tiles)"* — a sentence that **contradicts itself inside its own
parenthesis**, and which is very likely where the "blocks vs tiles" ambiguity came from in the first
place. Fixed: the three-layer table now has **Half-block** in it as a first-class row, `tiles.md`
carries the `& 1` proof, and the full unit table + proofs live in
[`reference/map-storage-locations.md`](../../reference/map-storage-locations.md) → "The units".
**Rename, don't resize.**

## 🧰 ONE Qt everywhere + the container's three "failures" were a lie in the plumbing

Twilight, on the two things left hanging from the CI-green session: *"i just hate the idea of holding
my program back because online is doing it … so sure i guess if you think its smart and not going to
mess things up itd be nice to fix ASan stuff locally here. idk go figure it out, help me."*

### 1. The container: 3 failures, not 13 — and the recorded cause was **backwards**

`status.md` said **13/91 fail** on environment: 24 × fontconfig + *"`tst_emu_parity` /
`tst_sound_parity` / `tst_flag_scenarios` need the ROM **+ PyBoy**, which the container lacks."*
Ran it before believing it. **88/91. Three failures. Zero fontconfig lines** — that class was already
gone; the count was stale from the months the image was unbuildable.

And the three were the **exact opposite** of "the container lacks the ROM + PyBoy":

```
FAIL!  : TstSoundParity::engineMatchesTheConsole(TrainerBattle) 'rc == 0' returned FALSE.
         (the emulator could not play this track:
          <3>WSL (3483 - ) ERROR: UtilGetPpid:1330: Failed to parse: /proc/1/stat …)
```

`docker/run-tests.sh` rsyncs the repo into the container excluding only `.git/` and the build dirs —
so the host's `tmp/` rode along, **Windows `emu-venv` and all**. The skip-gate was
`QFile::exists("tmp/emu-venv/Scripts/python.exe")`, hardcoded to the Windows layout in **three
copies**. On Linux that file *was there*, so the gate answered **available**, the QSKIP never fired,
and the test tried to `exec` a **Windows PE binary**. WSL's binfmt interop grabbed it and failed
sideways — which surfaced as red tests that looked like a code fault and weren't. **CI was green
precisely because a fresh checkout has no `tmp/`.**

⭐ **The lesson, and it is bigger than this bug:** *a file existing is not a capability being
available.* An availability gate that tests for a **path** instead of the **capability** will pass in
exactly the environment it was written to protect. It held only because the one machine that had a
venv was the one that made it.

**Fixed twice, deliberately** — belt *and* braces, because either alone leaves the trap set:

- **The gate is now runnability-based**: new shared `projects/tests/helpers/emuvenv.h`, used by all
  three tests. Platform-correct path (`Scripts/python.exe` ↔ `bin/python3`) and then the only
  question that matters — **`pythonRuns()`**: spawn `--version`, demand exit 0, cache it.
- **`run-tests.sh` excludes `tmp/`** — host scratch (screenshots, emu output, venv) has no business
  in the container regardless.

**Proven independently**, because otherwise the exclusion would hide whether the gate works. Re-injected
the Windows venv into the container by hand and ran the test:

```
--- windows venv IS present in container ---
SKIP   : TestEmuParity::initTestCase() emulator verification unavailable: tmp/emu-venv holds an
         interpreter that will not run on this platform (a venv built for another OS)
         -- rebuild it with scripts/emu/setup.ps1
```

A skip that **explains itself**, where there used to be WSL garbage.

**Green:** container **91/91 standard**, **91/91 asan**, **0 sanitizer findings, 0 fontconfig**.
Windows kit: the three emu tests still **pass for real** (the venv runs here) — the gate had to keep
working in the positive case, not just the negative one.

### 2. The Qt gap: closed upward — after I told her it couldn't be, and she didn't buy it

Options put to her: bump CI to 6.11 · drop the kit to 6.8.3 · keep the split + guard. **She chose to
bring the shipping toolchain UP** (*"i just hate the idea of holding my program back because online is
doing it"*). Pushed `feature/qt-6.11-alignment` with all five files at `6.11.0`.

**The branch run earned its keep immediately.** `lint` (Linux 6.11) **succeeded**. `tests` **died in
35 s** at the Windows `Install Qt` step, having compiled nothing:

```
ERROR : Failed to download checksum for the file 'Updates.xml' from mirrors '['https://download.qt.io']
```

Diagnosed exactly — and the diagnosis was right: **Qt reorganised the Windows repo at 6.11.** Old
layout nests the version folder once (`qt6_683/qt6_683/`, `qt6_6100/qt6_6100/`); 6.11 splits it
**per arch** (`qt6_6110/qt6_6110_llvm_mingw/` + 3 siblings, each with its own `Updates.xml`, and
nothing at the parent level). aqt asks for `qt6_6110/qt6_6110/Updates.xml` → 404. Probed: 6.9 and 6.10
install fine, **6.11.0 / 6.11.1 / 6.12.0 all fail identically**, same connection, same minute. Linux
untouched.

**Then I got it wrong.** PyPI says aqtinstall 3.3.0 is newest → *"no fix exists"* → wrote it up as a
**ceiling** ("Windows can have at most 6.10.x") and handed leadership three options built on top of
it: 6.10.3 everywhere, stay on 6.8.3, or put her Qt account password in GitHub secrets.

🐺 **She refused the premise:** *"i find it hard to believe the community has no solution for this
moving forward."* She was right, and it took ninety seconds to prove:

- issue **[#959](https://github.com/miurahr/aqtinstall/issues/959)** — the per-arch split, described
  exactly as diagnosed here, independently.
- PR **[#1000](https://github.com/miurahr/aqtinstall/pull/1000)** *"Support Qt 6.11+ for Windows
  x64"* — **MERGED 2026-03-24. Four months ago.**
- `install-qt-action` has an **`aqtsource`** input — *"Location to source aqtinstall from in case of
  issues"* — which exists for exactly this.

**The trap: 3.3.0 is from JUNE 2025.** The project had moved a year without cutting a tag, so "latest
release" said nothing about what the code could do. Verified the fix in a throwaway venv before
touching a workflow (`win64_llvm_mingw` listed at 6.11.0; all three modules present), then pinned it —
**to the merge commit, not `@master`**, because a moving master would make the job non-reproducible,
which is the very disease being cured. Windows jobs only.

So: **`6.11.0` everywhere** — kit == container == CI — with a ⚠️ block atop `tests.yml` recording what
the gap cost and that the five files move together, and a ✅ marker on the `aqtsource` lines saying to
delete them the day aqtinstall releases past 3.3.0.

⭐ **Three lessons, all in [`decisions/rejected.md`](../../decisions/rejected.md):** *"the latest
release can't" is not "can't"* · *a big ecosystem having no answer is a **smell**, not a finding* ·
*don't promote an unchecked constraint into a decision for someone else* — that one nearly bought a
permanent wrong answer (6.10.3, or credentials in CI) on the strength of one lazy step. Note the
cousin in §1: *a file existing is not a capability being available.* **Both are: check the thing, not
a proxy for it.** And note it cuts both ways — the branch run was right to be insisted on (it caught a
real break), and the write-up of what it found was still wrong.

`VERSION` does **not** move (test/CI-only, per the rule). The `QT_VERSION_CHECK(6,9,0)` guard in
`mapengine.cpp` **stays** — four lines, proven on both paths, keeps 6.8 source-buildable.

### 1. A heap-use-after-free in the music parser (the serious one)

`Gen1MusicAsm::parse` → `addrOf()` took **`const QString& label`**, and the caller passes a reference
**straight into `pending`**:

```cpp
auto addrOf = [&](const QString& label) -> int {   // <- reference INTO `pending`
  ...
  if (!placed.contains(file))
    place(file);                                   // <- place() does pending.append(...)  ==> REALLOC
  return placed[file] + streams[file].labels.value(label);   // <- `label` now dangles. READ of freed.
};

for (int i = 0; i < pending.size(); i++)           // the loop's own comment: "`pending` GROWS"
  const int addr = addrOf(pending.at(i).second);
```

The growth was known — the comment says so. That the growth **frees the caller's reference** was not.
Taken **by value** now.

ASan reported it **7 times from 5 different tests** (`tst_audio`, `tst_music_asm`, `tst_acceptance`,
`tst_gui_fidelity`, `tst_qml_screens`) — all **one root cause**: everything that parses the sheet music
went through it. It never crashed, because freed memory usually still holds what you left there. That
is exactly what makes the class dangerous.

### 2–4. The clang-tidy gate (3 findings, all legitimate)

- **`MapModel::mapInd()` — null dereference.** Bare `map->curMap`. `map` **is** nullable and this class
  says so itself: the ctor null-checks every pointer it connects to, and `connectionRoomLeft()` /
  `saveConns()` both guard `map`. **That is precisely why the analyzer could prove the path** — the
  code's own guards are the evidence. Now guarded, returning 0 like its siblings.
- **`mapengine.cpp:662` — redundant expression, with a real bug hiding behind it.**
  `mapInd < FirstIndoorMapId && mapInd < NumCityMaps` — `NumCityMaps` (10) is far below
  `FirstIndoorMapId` (0x25), so the second **always implied** the first. The check that was *missing*
  is **`>= 0`**, on an expression that indexes `SgbCity`. Dropped the dead half, added the real one.
- **A `Strip` copied where a `const&` would do.**

### The reason none of this was caught here — BOTH safety nets were down

- **ASan does not run on the llvm-mingw kit.** The **container** is the only place it runs…
- **…and the container had been broken since `pse-audio` landed.** Its image installed Qt with
  `-m qtcharts` alone, so once the build needed Qt Multimedia every run died at **configure** —
  *"Could NOT find Qt6Multimedia"* — before compiling a line. `dtest.ps1 asan` had not run in a long
  time and said so only in a log nobody read.
- **The remote could not compile** (the 6.9-only `QImage::flipped`).

So the one local tool that would have caught the UAF **could not start**, and the remote that would
have caught it **could not build**. That is the whole story of how it survived twelve versions.

**Container fixed** (`b070f1a` + follow-up): `-m qtcharts qtmultimedia qtshadertools` (now matching
`tests.yml`'s `modules:` line, with a comment to keep them in step) plus **`libpulse0` /
`libpulse-mainloop-glib0`** — `libQt6Multimedia.so` carries a hard `DT_NEEDED` on `libpulse.so.0`, so
without it **every test that links audio fails at the link step**, though nothing here ever plays a
sound. GitHub's runner ships PulseAudio, which is why the workflows install no apt deps and never hit
it; a clean container must ask.

**Verified:** local `dtest.ps1 asan` now builds and runs the suite again — **0 ASan/UBSan findings** —
and CI's `linux-asan` job is **success** on the same commit. Full local `ctest` **91/91**.
