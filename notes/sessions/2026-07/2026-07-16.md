## →The remote build had been red since 0.29.0 — the next "ship" would have failed (0.41.8-alpha)

Found while running the mandated pre-ship CI check, straight after saying *"91/91 green, working tree
clean, everything pushed"*. The remote disagreed: **`lint` and `tests` both `failure`** on `dev`. Not
the events work — `git log -S` dates it to **`e00a9ff` (0.29.0-alpha)**, and **every** `dev` run since
has failed.

**The cause:** `mapengine.cpp` drew the player facing right by X-flipping the left sprite with
`QImage::flipped(Qt::Horizontal)` — **Qt 6.9+**. The kit here is **6.11**; CI pins **6.8.3**:
`error: no member named 'flipped' in 'QImage'`.

**Why it matters more than a red badge:** `release.yml` builds the same `appcore` **and also pins
6.8.3**, and it hadn't run since **`v0.16.6-alpha`** — *before* the break. The whole Events feature is
sitting on `dev` waiting for "ship", and **that ship would have merged to `main` and then failed to
build the downloads.** Caught by the check, one step before it mattered.

**The fix:** a `QT_VERSION_CHECK(6,9,0)` guard — `flipped()` on new Qt, `mirrored(true, false)` on old
(identical result; not deprecated until **6.13**, so unguarded `mirrored()` would just trade a CI error
for a local warning). One helper, both sprite paths.

**The verification worth copying:** the `#else` branch *never compiles here* — 6.11 always takes the
`#if` — so a typo in it would surface only as a second red CI round-trip. I forced the guard to
`(99,9,0)`, rebuilt, and **ran `tst_map` on the fallback**: `27 passed, 0 failed`, same as the 6.11
path. The branch the release pipeline actually compiles is proven compilable *and* pixel-identical
before the push, not after.

**The uncomfortable finding:** there is **no local way to catch this class at all**. Only Qt 6.11 is
installed, and the Docker image is 6.11 as well — its own Dockerfile says *"Mirrors project leadership's local
kit: Qt 6.11 + clang. (CI uses gcc/6.8.3…)"*. **The remote is this project's only 6.8 compiler**, which
is exactly why "green" includes CI and why local green was never sufficient here. Also: a failed CI
build stops after ~7 of 414 files, so more 6.9-isms may hide behind this one — pushed to find out.

Written up in [`reference/qt-patterns.md`](../../reference/qt-patterns.md) (top) +
[`reference/fix-patterns.md`](../../reference/fix-patterns.md). **Flagged for leadership, not decided:**
dev `6.11` vs shipping `6.8.3` is the root cause — align them, or hold the floor and keep guarding.

## Phase 6: the event model is byte-perfect — and mislabels 14 bits (Pokémon Tower)

Verified v2's event model (`WorldEvents` + `EventsDB`/`events.json`; data-driven per-event `byte`+`bit`)
against the cartridge-verified truth. Two findings, one good, one a real bug.

**✅ The byte/bit maths is flawless — no corruption.** All 508 entries satisfy `byte == 0x29F3 + ind/8`
and `bit == ind%8`: **0 mismatches**, all in range, no duplicates. `ind` **is** pret's event index, and the
offsets match the verified anchor exactly (`Followed Oak Into Lab` ind 0 → byte 10739 = `0x29F3` bit 0;
`Beat Articuno` ind 2522 → `0x2B2E` bit 2). Unlike the tileset `collPtr` case, **nothing writes a wrong
byte.**

**But 14 entries are MISLABELLED and 14 more are phantoms — all in the Pokémon Tower block**
(`0x0EE`–`0x113`), where v1's list is **shifted ~2 bits** off the truth. 480/508 names match pret; the
strays mean **the editor writes the wrong flag**:

- *"In Purified Zone"* → really `EVENT_BEAT_POKEMONTOWER_5_TRAINER_3`
- *"Beat Pokemontower 7 Trainer 0"* → really **`EVENT_BEAT_GHOST_MAROWAK`** (story-critical)
- *"Beat Pokemontower 3 Trainer 0"* → really `EVENT_BEAT_POKEMON_TOWER_RIVAL`
- The **14 phantoms** (`0x0EE`, `0x0EF`, `0x0F2`, `0x0F9`, `0x0FA`, `0x102`, `0x103`, `0x10A`, …) sit on
  **placeholder bits pret never names** → toggling them does **nothing**.
- (4 of the "mismatches" are benign style only — `Unnamed 1BF` vs `EVENT_1BF`, same bit. `0x677` is
  unnamed in v1 but is really `EVENT_ENTERED_ROCKET_HIDEOUT`.)

Leadership context (2026-07-16): the v1 event data *"was always a mess to me… but I never disputed it was
contiguous bytes"* — so this is legacy confusion, not a disputed model. Which also means the
`worldevents.h` comment (*"scattered all over the save"*) is **misleading wording** and gets fixed: it is
one **contiguous 320-byte array**.

**The fix is Phase 6b/7, not a hand-patch:** `ind` is pret's index and `import_event_flags.py` already
emits the canonical, self-validating 2,560 rows — so **regenerate `events.json` from pret**, which also
lifts coverage **508 → 2,560** (the other 2,052 are unreachable today). ⚠️ Data change → **awaiting
leadership sign-off** (no `.json` edits without it). Detail + table:
[`../../reference/event-flags.md`](../../reference/event-flags.md).

## ❌ The founding conflict — Route 22's two rivals — is REFUTED (console + source)

The event-flags **conflicting-flags system** (plan Phase 11) was founded on the Route 22 case: two
`SPRITE_BLUE` objects at the **same tile (25,5)** (`ROUTE22_RIVAL1`/`RIVAL2`, missables 34/35, one per
battle), so **both battle flags on "must" collide**. It was recorded `suspected-strong` on "overwhelming"
static evidence (2026-07-15). **It is wrong** — and catching that is worth more than the original claim.

**The source settles it** (`scripts/Route22.asm`, `Route22DefaultScript`) — an **ordered if/else**:

```
	CheckEvent EVENT_1ST_ROUTE22_RIVAL_BATTLE
	jr nz, Route22FirstRivalBattleScript     ; 1ST set -> FIRST battle, and DONE
	CheckEventReuseA EVENT_2ND_ROUTE22_RIVAL_BATTLE   ; only reached if 1ST is CLEAR
```

With 1ST set the 2ND is **never read** — **masked, not conflicting**; the stacked sprites merely overlap
and only one is ever driven. **The cartridge agrees** (`probe_route22_conflict.py`, ~5 s via the new MCP):

```
armed-1st -> HEALTHY trainer battle (mode=2, script=2)
conflict  -> HEALTHY trainer battle (mode=2, script=2)   <- both flags + both sprites shown
ghost     -> NO BATTLE (script=1, ow=True) - stalled/blocked
```

⚠️ **Trigger coords corrected:** the ambush fires from `.Route22RivalBattleCoords` = `dbmapcoord 29,4` /
`29,5` (**`dbmapcoord` stores db y,x**) — **not** the rival's own tile (25,5), which is what the 07-15
attempts kept driving at. The player must stand on the **upper** trigger (29,4); (29,5) blocks the rival's
rightward walk and softlocks the cutscene (a bad forge, not a game bug). Arming also needs
`wRoute22CurScript = DEFAULT` **and** the rival object shown — flags alone are not enough.

**The three rules this bought — the system's spine:**

1. **`suspected` is a LEAD, not a risk.** A same-subject cluster / two objects on one tile is a reason to
   *look*, never to *warn*. **Only `confirmed` renders a crash warning in the UI.**
2. **`refuted` is kept, not deleted** — negative knowledge; the static generator now **defers** to verdicts
   (`REFUTED_SUBJECTS`) so a settled question is never re-raised.
3. **Adjudicate on the console** before anything reaches a user — script **dispatch order** can mask a flag
   entirely, and no amount of static co-location proves a conflict.

**A REAL candidate fell out of it:** `route22-rival-armed-but-hidden` (**suspected/softlock**) — armed
(WANTS + 1ST) while the rival object is **hidden** → the trigger fires, the script advances to 1, and **no
battle engages** (stalled). A genuine **flag ↔ missable inconsistency** the static pass would never have
found; owed its own probe. (It also contradicts the probe docstring's "ghost engages cleanly too" — the run
says otherwise; **trust the run**.)

**Dataset now:** 7 rules — `route22-rival-overlap` **refuted/none**, `route22-rival-armed-but-hidden`
**suspected/softlock**, + 5 same-object governors (Snorlax ×2, Cerulean rival, Giovanni, Town Map), all
**suspected** with severity **unknown-pending-adjudication** (no longer auto-labelled "crash"). Corrected
in `analyze_flag_contradictions.py`, [`../../reference/event-flags.md`](../../reference/event-flags.md),
and [`../../plans/event-flags.md`](../../plans/event-flags.md) → Phase 11.

**Tooling note:** the `pokered-dev` MCP server (job-based, single-shot, tree-killed, `procs_cleanup`) is
what made this adjudication possible — the same probe that fought the interactive shell for hours on 07-15
now runs in **~5 s**. It is the standing transport for emulator work.

## Forged saves at ANY map — the console authors them itself (project leadership's brief, same day)

Project leadership briefed the full thing mid-session: the MCP server must generate **proper save files at any
map state** — tileset/blockset right, sprites right, "the pointers right, lots of little calculations

  The generator therefore reaches outdoor targets from INSIDE Red's house (tileset 1), leaving via the

## The PyBoy hang SOLVED, the dev MCP server built, the debug system tightened

Project leadership opened with the unexplained disaster: PyBoy runs "spawning tons of processes and hanging,"
the previous session unable to get a clean automated save-file test. Diagnosed to the root, fixed at
every layer, and the whole dev workflow got the standardized transport they asked for.

### The root cause, finally — THREE stacked problems, all reproduced

1. **A map-id-only forge is a CHIMERA and it hard-wedges the console.** Forging just
   `wCurMap`(+coords) onto another map leaves the save's whole Area block (dims/tileset/scripts/
   sprites) belonging to the *old* map — and the `BIT_NO_PREVIOUS_MAP` linchpin means Continue TRUSTS
   it. The console really enters the map (emulator: `map=0x21, w=10, h=9` — Route 22 wearing Pallet's
   body) and **crashes within ~100 frames**: the runaway CPU executes `STOP`, the clocks halt, the
   frame never completes, and **`pb.tick()` never returns** — a full core, forever (the glitch-music
   bad-bank probe recorded the same console behaviour). Isolated cleanly: `forge`-only and `mash`-only
   both run 6,000 frames in ~0.5 s; `forge+mash` wedges after Continue. **This overturns 2026-07-15's
   "forge-onto-any-map PROVEN"** — `on_overworld()` is true for a moment, then the crash lands; the
   probe had stopped looking one moment too soon. Same-map forges (flags/coords/any Main-Data byte)
   remain fully consistent and safe.
2. **Killing the wedged child leaked its interpreter.** The venv `python.exe` is a *launcher* whose
   child is the real interpreter: a plain kill (QProcess::kill, subprocess.kill) can take the launcher
   and orphan the spinning PyBoy. Kill the **tree** (`taskkill /T /F`), always.
3. **The interactive shell's ~44 s cap made both unrecoverable** (the 2026-07-15 diagnosis, confirmed):
   management commands timing out mid-kill leaked more, starved the CPU, timed out more.
